# Seven: HROps — 产品功能说明文档 v5.0

> **HR 的 Claude Code / HR 的 OpenClaw**
>
> 本地优先的 HR 个人智能工作台——把 HR 每天最耗时的"文档处理 + 提纲设计 + 访谈分析 + 报告撰写"全部 AI 化

---

## 📋 一、产品定位

### 核心定位

```text
Claude Code 之于程序员 = Seven: HROps 之于 HR
```

Claude Code 把程序员日常的"读代码、写代码、调试、文档"Agent 化。
Seven: HROps 把 HR 日常的"筛简历、出提纲、做分析、写报告"Agent 化。

### 产品核心逻辑

```text
┌─────────────────┐     ┌──────────────────────┐     ┌─────────────────┐
│   HR 输入数据    │ ──→ │    AI 处理引擎         │ ──→ │   输出给 HR     │
│                 │     │                      │     │                 │
│ · 拖拽简历文件   │     │ · 简历解析 + 评分      │     │ · 初筛报告      │
│ · 粘贴 JD 文本  │     │ · JD 结构化           │     │ · 面试提纲      │
│ · 选择访谈场景   │     │ · 胜任力提取           │     │ · 访谈分析报告  │
│ · 粘贴访谈记录   │     │ · STAR/冰山/BEM 分析  │     │ · 员工画像      │
│ · 录入面试评价   │     │ · 提纲生成            │     │ · 合规报告      │
│ · 邮箱简历收取   │     │ · 合规检查            │     │ · 话术文本      │
└─────────────────┘     └──────────────────────┘     └─────────────────┘
```

**一句话**：HR 把数据喂进来，AI 处理，输出结果给 HR。软件不替 HR 跟人交互，只替 HR 做脑力劳动。

### 🔑 核心设计诉求

#### 1. 固定工作流中的 LLM 调用必须走缓存

> **背景**：产品中存在大量"固定工作流"——即每次执行步骤相同、输入相同时，大模型的回答也必然相同的场景（如：对同一份简历做同一个 JD 的评分、对同一段访谈记录做同一维度的分析）。这类重复请求不应每次都真实调用 LLM。

**核心原则**：

```text
相同输入 + 相同工作流步骤 = 直接返回缓存结果，不重复请求大模型
```

**适用场景**：

| 场景 | 说明 |
|------|------|
| 简历评分 | 同一份简历 + 同一个 JD → 评分结果缓存，不重复调用 |
| JD 结构化解析 | 同一段 JD 文本 → 结构化结果缓存 |
| 面试提纲生成 | 相同 JD + 相同候选人简历 → 提纲缓存 |
| 访谈分析 | 相同访谈记录 + 相同分析维度 → 分析报告缓存 |
| 合规检查 | 相同文本 + 相同合规规则集 → 检查结果缓存 |

**缓存 Key 设计原则**（待实现时细化）：

- Key = `hash(工作流类型 + 输入内容 + 使用的 Skill/Prompt 版本)`
- 输入内容变化（如简历被重新上传、JD 被修改）→ 缓存失效，重新请求
- Skill/Prompt 版本升级 → 缓存失效，重新请求
- 缓存存储位置：本地 SQLite（与业务数据同库，独立表）

**实现时机**：

> ⚠️ **当前阶段暂不实现**。此诉求已记录在文档中，待对接 OpenCode Sidecar 时统一处理。
> 对接 OpenCode 时需在 SDK 调用层做拦截：请求前查缓存，命中则直接返回；未命中则调用 LLM 并将结果写入缓存。

### ❌ 明确不做的事

| 不做 | 原因 |
|------|------|
| 替 HR 跟候选人聊天 | 你不是微信/邮件客户端，没有候选人联系方式 |
| 自动安排面试排期 | 不知道面试官是谁、日程是什么、候选人时间偏好 |
| 自动发送提醒通知 | 没有面试官和候选人的通讯录 |
| 爬取招聘网站简历 | 招聘网站有反爬机制，这是爬虫不是工具 |
| 社交网络人才寻访 | 没有脉脉/LinkedIn API 权限 |
| 竞争对手薪酬分析 | 没有外部数据源，本地只有 HR 自己录入的数据 |
| 候选人端产品 | 你是 HR 工作台，不是招聘平台 |

## 🔧 四、功能模块详细规划

### 模块 1：简历收取与管理

> **痛点**：简历来源分散（邮件附件、网站下载、HR 自己保存），整理归档耗时。

#### 1a. 邮箱简历收取（开源版核心差异化功能）

**工作原理**：HR 配置自己的工作邮箱 → 系统定时扫描 → 识别简历附件 → 自动下载解析 → 归档到对应 JD

**支持邮箱**：
- QQ 邮箱 / 网易 163/126（IMAP/POP3）
- Gmail / Outlook（IMAP）
- 企业邮箱（腾讯企业邮、阿里企业邮、通用 IMAP）

**核心能力**：
- 📧 定时扫描邮箱（可设置间隔：15分钟/1小时/手动）
- 📎 识别并下载简历附件（PDF/Word/图片）
- 🏷️ 从邮件主题/正文提取候选人基本信息
- 📂 自动归档到对应 JD（基于邮件内容匹配）
- 🔔 新简历桌面通知

**HR 操作流程**：
```
设置 → 邮箱配置 → 填写 IMAP 信息
  → 选择扫描规则（发件人/主题关键词）
  → 开启自动扫描
  → 新简历到达 → 桌面通知
  → 进入简历库查看
```

#### 1b. 本地文件导入

- 拖拽 PDF/Word/图片简历到应用窗口
- 选择文件夹批量导入
- 支持格式：PDF / DOCX / DOC / JPG / PNG

#### 1c. 简历库管理

- 按 JD 分类管理简历
- 候选人生命周期状态跟踪（待筛选 → 已筛选 → Shortlist → 待面试 → 已面试 → 评估中 → 待Offer → 已Offer）
- 简历搜索（姓名/技能/公司/学校/来源）
- 简历去重检测

### 模块 2：JD 管理

> **痛点**：JD 文本散落在各处，每次招聘都要重新整理，JD 与简历的对应关系不清晰。

**核心能力**：
- 📋 JD 文本粘贴 → AI 自动结构化解析（岗位名称/职责/要求/薪资范围）
- 🏷️ 提取核心胜任力要素（用于简历匹配和面试提纲生成）
- ✏️ JD 模板库管理（常用 JD 保存复用）
- 🔗 JD 与简历的关联管理（一个 JD 对应多份简历）
- 📊 JD 招聘进度看板（收到多少简历/通过多少/面试多少/Offer 多少）

---

### 模块 3：简历初筛 Agent ⭐核心

> **痛点**：200 份简历人工筛选要 4-8 小时，主观性强，容易遗漏，标准不统一。

**工作原理**：HR 选择 JD → 选择待筛简历 → AI 自动评分排序 → HR 查看报告确认

**评分维度**（基于冰山模型 + 胜任力模型）：

```
简历评分卡
├── 硬性条件（40%）
│   ├── 学历匹配度
│   ├── 工作年限
│   ├── 必备技能关键词
│   └── 资格证书
├── 软性条件（30%）
│   ├── 行业经验相关性
│   ├── 项目经历相关性
│   └── 公司背景（知名度/规模/行业）
├── 亮点加分（20%）
│   ├── 知名企业经历
│   ├── 特殊成就/奖项
│   └── 额外技能
└── 合规检查（10%）
    ├── 反歧视合规
    └── 信息完整性
```

**输出**：
- 每份简历的评分卡（含评分理由，可追溯）
- 简历排序列表（按匹配度降序）
- 初筛报告（通过/不通过 + 理由）
- 批量导出报告（PDF/Excel）

**合规设计**：
- 初筛阶段可选择隐藏姓名/性别/年龄/照片（去偏见模式）
- 所有 AI 决策留有证据链，可追溯

### 模块 4：面试提纲生成 Agent ⭐核心提效

> **痛点**：面试官没准备，问的都差不多；HR 出提纲要 1-2 小时；不同面试官标准不一。

**基于**：`employee-interview-generator` SKILL 的核心能力，内置到桌面端。

#### 4a. 招聘面试提纲（含技术面试）

**输入**：JD 信息 + 候选人简历 + 面试类型（初面/复面/终面/技术面/HR面）

**AI 处理**：
1. 从 JD 提取核心胜任力维度
2. 分析简历亮点和疑点
3. 按 STAR 法则生成行为面试问题
4. 按岗位技术栈生成技术面试问题
5. 生成针对简历疑点的追问
6. 为每个问题生成评分量表（1-5 分 + 行为锚点描述）

#### 4a-1. 技术面试问题深度生成 ⭐新增

> **痛点**：技术面试官出题随意，问的问题跟岗位需求不匹配；HR 无法判断技术面试质量。

**输入**：JD 技术栈要求 + 候选人简历中的技术经历 + 面试难度（初级/中级/高级/专家）

**AI 处理**：
1. 从 JD 提取技术栈要求（语言/框架/中间件/架构）
2. 从简历提取候选人技术经历（项目/技术深度/技术广度）
3. 交叉分析 JD 要求 vs 简历经历 → 找出需要验证的技术点
4. 按难度梯度生成技术问题（基础→进阶→深度→系统设计）
5. 为每道题生成参考答案要点 + 评分标准
6. 生成追问链（候选人答对→追问更深；答错→换角度验证）

**技术面试提纲结构**：
```
📋 技术面试提纲：高级后端工程师 - 王五（技术面 · 60分钟）

一、技术基础验证（15分钟）
  【Java 核心】（JD 要求：Java 5年+）
  Q1: "HashMap 在并发场景下会出什么问题？ConcurrentHashMap 怎么解决的？"
      难度：⭐⭐ 中级
      参考要点：死循环(JDK7)/数据丢失 → 分段锁→CAS+synchronized
      评分：1分=不知道/3分=知道问题/5分=能讲底层实现
      追问链：
        → 答对："JDK8 的 ConcurrentHashMap 和 JDK7 有什么区别？"
        → 答错："那你在项目中遇到过并发问题吗？怎么处理的？"（转STAR）

  Q2: "JVM GC 调优你做过吗？讲一个具体案例"
      难度：⭐⭐⭐ 高级
      参考要点：GC 算法选择/参数调优/监控工具/实际效果
      评分：1分=只知道概念/5分=有实战经验+量化结果

二、项目技术深度（20分钟）
  【简历项目深挖】（简历亮点：主导过千万级用户系统重构）
  Q3: "你主导的系统重构，技术方案是怎么设计的？"
      追问：为什么选这个方案？考虑过哪些替代方案？
      追问：重构过程中最大的技术挑战是什么？
      追问：上线后的效果？有没有量化数据？

三、系统设计（20分钟）
  Q4: "设计一个支持 10 万 QPS 的短链接服务"
      难度：⭐⭐⭐⭐ 专家
      考察点：哈希算法/存储选型/缓存策略/高可用/扩展性
      评分维度：需求分析/架构设计/技术选型/边界处理/扩展思考

四、技术视野（5分钟）
  "最近在关注什么新技术？有没有在项目中尝试？"
```

**支持的技术栈领域**：
- 后端：Java/Go/Python/Node.js/C++/Rust
- 前端：React/Vue/Angular/TypeScript/性能优化
- 移动端：iOS(Swift/ObjC)/Android(Kotlin/Java)/Flutter/RN
- 大数据：Spark/Flink/Kafka/Hadoop/数据仓库
- AI/ML：PyTorch/TensorFlow/NLP/CV/推荐系统
- 基础设施：K8s/Docker/CI-CD/监控/云原生
- 数据库：MySQL/PostgreSQL/Redis/MongoDB/Elasticsearch

**输出示例**：
```
📋 面试提纲：高级前端工程师 - 张三（初面 · 45分钟）

一、破冰开场（3分钟）
  话题建议：最近关注的前端技术趋势

二、核心能力考察（25分钟）
  【技术深度】
  Q1: "讲一个你解决过的最复杂的前端性能问题"（STAR行为面试）
      评分锚点：1分=只描述问题/5分=完整STAR+量化结果
      追问：性能提升的 baseline 是什么？怎么测量的？
  Q2: "如何设计一个高性能的虚拟滚动组件？"（技术深度）
      评分锚点：1分=知道概念/5分=能讲原理+边界处理+优化方案

  【项目经验】
  Q3: "你主导的最有挑战的项目是什么？"（STAR）
      追问：你在团队里的角色？遇到的最大阻力？

三、亮点与疑点深挖（10分钟）
  ⭐ 亮点：5年字节跳动经验
     → "字节的工作节奏和你现在的期望有什么不同？"
  ⚠️ 疑点：简历中有1年空白期（2024.3-2025.3）
     → "那段时间你在做什么？"

四、文化与动机（5分钟）
  "为什么离开现在的公司？"
  "你对新团队有什么期待？"

五、候选人提问（2分钟）
  预判问题：薪资/远程政策/技术栈选择
```

### 模块 5：访谈记录智能分析 Agent ⭐核心

> **痛点**：访谈完了不知道怎么分析，写报告要 2-3 小时，不同 HR 分析质量差异大。

**基于**：`employee-interview-analyzer` v2.12.0 SKILL 的核心能力，内置到桌面端。

**输入**：粘贴访谈记录文本（或上传文档）

**AI 处理（13步分析流程）**：
1. 智能场景识别（自动识别是哪种访谈）
2. 文本修正（错别字/语句不通）
3. 方法论自动匹配（≤3种，优先2种）
4. 展示分析框架（HR 确认或调整）
5. 多维度分析（假设驱动 + 框架分析）
6. 交叉验证（矛盾检测）
7. 洞察生成（结论 + 建议 + 行动）
8. 画像生成（三轨策略）
9. 组织画像（仅组织诊断场景）
10. 可选：数据可视化
11. 可选：结论文章（800-1500字）
12. 输出完整报告

**7 大方法论框架**（对标 `employee-interview-analyzer`）：

| 方法论 | 适用场景 | 核心分析能力 |
|--------|---------|-------------|
| 杨三角理论 | 组织诊断/团队反馈 | 员工能力/思维/治理三支柱诊断 |
| 韦斯伯德六盒 | 组织诊断 | 六维度系统评估组织健康度 |
| 推拉力理论 | 离职/挽留访谈 | 推力/拉力/留任力/移动障碍分析 |
| 冰山模型 | 招聘/晋升/入职 | 知识/技能/角色/认知/特质/动机六层评估 |
| 吉尔伯特 BEM 模型 | 绩效/入职/发展 | 环境(75%)/个人(25%)因素分析绩效差距 |
| 职业锚 + GROW 模型 | 职业发展/辅导 | 职业锚识别 + Goal/Reality/Options/Will |
| STAR 面试法 | 招聘面试 | Situation/Task/Action/Result 行为评估 |

**三轨画像系统**：
- 离职访谈 → 跳过画像，聚焦离职原因分析
- 招聘面试 → 候选人综合画像（含岗位匹配分析 + STAR 场景呈现）
- 其他场景 → 管理干部画像 or 员工画像（自动识别管理者身份）

**管理干部画像包含**：
- 基础画像（工作投入度/异动风险/战场/战场匹配度/风险与挑战/优劣势/发展诉求）
- 多场景管理表现（业务管理/团队管理/周边协作/资源使用/自我管理）
- 行为记录按 STAR 原则，含发生时间

### 模块 8：面试评价自动生成 Agent ⭐新增核心

> **痛点**：面试结束后，面试官要花 30-60 分钟写评价；评价散落在各处（纸质/邮件/IM）；多面试官评价难以汇总；标准不统一。

> **核心思路**：面试过程中 HR 录音或记录文字 → 面试结束后导入系统 → AI 自动生成结构化面试评价。

#### 8a. 面试过程记录导入

**支持的输入方式**：

| 输入方式 | 说明 | 适用场景 |
|---------|------|----------|
| 录音文件上传 | 上传 MP3/WAV/M4A 录音文件 → 自动转文字 | 线下面试（需候选人同意录音） |
| 文字记录粘贴 | 直接粘贴面试过程的文字记录 | HR 手动记录/IM 面试记录 |
| 实时语音转写 | 面试过程中开启实时转写（需麦克风权限） | 线上/线下面试 |
| 面试笔记导入 | 导入面试官的简要笔记 | 面试官只记了要点 |

**录音转文字技术方案**：
- 本地转写：Whisper（OpenAI 开源，支持离线，保护隐私）
- 云端转写：讯飞/阿里云 ASR（可选，更准确）
- 说话人分离：自动区分面试官和候选人的发言

#### 8b. AI 自动生成面试评价

**工作原理**：面试记录（文字）+ 面试提纲（如果有）+ JD + 简历 → AI 分析 → 结构化评价报告

**AI 处理流程**：
1. **对话结构化**：识别面试官提问 vs 候选人回答，按问题分段
2. **STAR 行为提取**：从候选人回答中提取 Situation/Task/Action/Result
3. **能力维度评估**：按 JD 胜任力维度逐项评分（1-5 分）
4. **技术深度评估**：评估候选人技术回答的深度和准确性
5. **亮点与风险标注**：标记候选人的突出表现和潜在风险
6. **综合评价生成**：生成结论性评价 + 录用建议

### 模块 11：SKILL 插件系统 ⭐新增核心

> **核心理念**：Seven: HROps 的能力不应该是固定的，而是可以通过 SKILL 插件扩展和调整的——就像 Claude Code 的 SKILL 一样。

> **架构决策**：SKILL 系统基于 OpenCode 的 Skill 机制实现。OpenCode 原生支持 `.opencode/skills/` 目录下的 SKILL.md 文件发现和加载，我们在此基础上扩展 HR 专属的 SKILL 生态。

> **对标**：Claude Code 的 `.claude/skills/` 机制 + OpenCode 的 `.opencode/skills/` 机制 + OpenClaw 的 SKILL 生态。

#### 11a. 什么是 SKILL

SKILL 是一个 **Markdown 格式的能力描述文件**，定义了一个 Agent 的行为规则、处理流程、输出格式等。Seven: HROps 的每个核心 Agent 本质上都是一个内置 SKILL。

**SKILL 文件结构**（基于 OpenCode Skill 格式，存放于 `.opencode/skills/` 目录）：

```yaml
---
name: my-custom-skill           # SKILL 唯一标识
description: 自定义的XX分析技能    # 一句话描述
author: hr-zhangsan              # 作者
version: 1.0.0                   # 版本号
tags: ["hr", "custom"]           # 标签
base_skill: interview-analyzer   # 基于哪个内置 SKILL 扩展（可选）
---

# SKILL 正文（Markdown）

## 何时使用
触发条件描述...

## 处理流程
1. Step 1: ...
2. Step 2: ...

## 输出格式
模板定义...

## 约束条件
规则和限制...
```

**与 OpenCode Skill 的关系**：
- OpenCode 自动发现并加载 `.opencode/skills/` 下的所有 `SKILL.md` 文件
- 每个 SKILL 对应一个 OpenCode 自定义 Agent 的行为规则
- SKILL 中可引用 MCP 工具（如 `seven-hrops_list_resumes`）
- SKILL 支持参数覆盖（通过 `.opencode/overrides/` YAML 文件）

#### 11b. SKILL 的三种使用方式

| 使用方式 | 说明 | 示例 |
|---------|------|------|
| **安装社区 SKILL** | 从 SKILL 市场下载安装 | 安装「大厂 HRBP 离职分析 SKILL」 |
| **自定义 SKILL** | HR 自己编写 SKILL 文件 | 编写适合自己公司的绩效面谈 SKILL |
| **调参覆盖** | 不改 SKILL 逻辑，只覆盖某些参数值 | 把初筛评分的「学历权重」从 40% 改为 20% |

#### 11c. 内置 SKILL 列表（可被覆盖/扩展）

| 内置 SKILL | 对应模块 | 可调参数示例 |
|-----------|---------|-------------|
| `screener` | 简历初筛 Agent | 评分维度权重、硬性条件阈值、去偏见规则 |
| `interviewer` | 面试提纲生成 Agent | 面试时长、问题数量、方法论偏好、技术栈范围 |
| `analyzer` | 访谈记录分析 Agent | 方法论优先级、画像维度、分析深度 |
| `evaluator` | 面试评价生成 Agent | 评估维度、评分标准、录用建议阈值 |
| `communicator` | 话术生成 | 话术风格、渠道适配规则、语气偏好 |
| `compliance` | 合规检测 Agent | 歧视词库、PII 规则、法律条款引用 |

#### 11d. 调参覆盖机制（Override）

> **场景**：HR 不想写完整 SKILL，只想调整某个 Agent 的某些参数。

**覆盖文件格式**（`.opencode/overrides/screener.yaml`）：

```yaml
# 覆盖简历初筛 Agent 的默认参数
skill: screener
overrides:
  # 调整评分权重
  scoring_weights:
    hard_requirements: 0.30    # 默认 0.40，降低硬性条件权重
    soft_requirements: 0.35    # 默认 0.30
    highlights: 0.25           # 默认 0.20
    compliance: 0.10           # 默认 0.10

  # 调整硬性条件
  hard_requirements:
    min_experience_years: 3    # 默认不限
    required_education: null   # 不限学历（覆盖默认的本科要求）
    must_have_skills:          # 必须包含的技能关键词
      - "React"
      - "TypeScript"

  # 去偏见模式
  bias_free_mode: true         # 隐藏姓名/性别/年龄/照片
```

**覆盖面试提纲生成参数**（`.opencode/overrides/interviewer.yaml`）：

```yaml
skill: interviewer
overrides:
  # 面试提纲偏好
  default_duration: 60         # 默认面试时长（分钟）
  questions_per_dimension: 3   # 每个维度的问题数
  include_scoring_rubric: true # 是否包含评分量表
  include_follow_ups: true     # 是否包含追问链

  # 技术面试偏好
  tech_interview:
    difficulty_distribution:   # 难度分布
      basic: 0.2               # 20% 基础题
      intermediate: 0.4        # 40% 中级题
      advanced: 0.3            # 30% 高级题
      expert: 0.1              # 10% 专家题
    include_system_design: true
    include_coding: false      # 不出编码题（面试环境不适合）

  # 方法论偏好
  preferred_methodologies:
    - "STAR"                   # 优先使用 STAR
    - "冰山模型"               # 其次冰山模型
```

#### 数据流

1. **用户在 AI Chat 输入消息** → 前端通过 `@opencode-ai/sdk` 发送到 OpenCode Sidecar
2. **OpenCode Agent 解析意图** → 选择合适的 Agent（Screener/Compliance/General）
3. **Agent 调用 MCP 工具** → 通过 MCP Protocol 调用 `seven-hrops` MCP Server
4. **MCP Server 转发** → 调用 Rust 命令层（`invoke()`）
5. **Rust 执行 CRUD** → 操作 SQLite / 文件系统
6. **结果回流** → MCP Server → OpenCode Agent → 流式响应 → 前端 UI

### D. 版本迭代路线图（V1~V5）

> 整合自原 `Series.md`，定义产品从单点提效到 HR Agent 平台的远期演进方向。
> V1 详细功能以本文档 §四 功能模块和 §七 开发路线图为准。

#### 核心迭代逻辑

> **招聘 → 招聘运营 → 员工运营 → 绩效管理 → 组织智能**

招聘天然是切入点：文档密集、决策频繁、AI 可直接介入、单人可使用、桌面端场景强、ROI 明显。

不能一开始做全 HR 平台的原因：
1. 没有单点价值爆发——用户不会因为"覆盖很广"买单，而是因为某个场景省了真时间
2. 数据结构太散——招聘/绩效/组织数据模型完全不同
3. Agent 能力会虚——什么都做往往只是"很多 tab 的后台"
4. 桌面端优势发挥不出来——第一阶段适合文档处理、本地数据、批量分析，不是全员协作系统

#### V1：Recruiting MVP — 看简历

**定位**：AI-powered desktop screening agent for recruiters

**核心功能**：JD 管理 → 简历导入解析 → AI 筛选评分 → 候选人对比 → 导出

**北极星指标**：让 HR 花更少时间完成初筛
- 衡量：单个职位筛选时间缩短多少；shortlist 产出速度提升多少

**目标结果**：证明用户愿意把真实简历扔进来；证明 AI 评分和对比有价值；跑通桌面端 + 本地 AI + 文档处理链路

**预计周期**：2~3 个月

#### V2：Recruiting Ops — 做招聘

**定位**：AI agent workspace for recruiting operations

从"筛选工具"升级成"招聘流程中的 AI 工作代理"。V1 解决"看简历"，V2 解决"做招聘工作"。

**核心功能**：
1. **职位工作台**：多职位项目管理、JD 模板库、岗位版本管理
2. **人才库管理**：候选人池、去重、标签体系、历史候选人检索
3. **候选人沟通辅助**：自动生成沟通话术、面试邀请/拒信模板、跟进提醒
4. **面试流程辅助**：面试轮次记录、面试反馈总结、多面试官反馈聚合、AI 总结候选人优劣势
5. **招聘分析**：渠道效果、筛选漏斗、职位处理效率、shortlist 转化情况

**Agent 形态演进**：引入多个 HR Agent 角色——Sourcer Agent / Screener Agent / Coordinator Agent / Interviewer Summary Agent / Recruiting Analyst Agent

**北极星指标**：让招聘流程中的重复劳动减少
- 衡量：沟通文案生成率、面试总结效率、招聘项目流转速度

**预计周期**：2~4 个月

#### V3：People Ops / Employee Ops — 管员工

**定位**：AI operations layer for employee lifecycle work

从招聘扩展到员工入转调离和 HR 日常运营——从"候选人"进入"员工"。

**核心功能**：
1. **员工档案与信息工作台**：员工基础档案、任职信息、团队归属、历史变更记录
2. **入职与试用期**：入职清单生成、Offer/入职文档检查、试用期跟进提醒、转正评估辅助总结
3. **HR 文档与制度助手**：政策问答、员工手册解析、假勤/福利/制度说明生成、内部通知拟稿
4. **员工异动管理**：调岗记录、晋升建议归档、离职流程辅助、离职原因总结
5. **员工沟通辅助**：1on1 纪要整理、HRBP 跟进记录、员工问题摘要

**Agent 形态**：Onboarding Agent / People Ops Agent / Policy Agent / HRBP Assistant / Employee Record Agent

**北极星指标**：让 HR 日常事务处理更轻
- 衡量：入转调离处理时间、政策解释与文档生成效率、HRBP 记录整理效率

**预计周期**：3~5 个月

#### V4：Performance & Org Intelligence — 做绩效和组织分析

**定位**：AI intelligence layer for performance and organizational capability

从"提效工具"走向"智能决策平台"。

**核心功能**：
1. **绩效管理辅助**：绩效评语生成辅助、自评/他评总结、绩效校准会纪要总结、绩效偏差提醒、绩效趋势概览
2. **能力模型与胜任力**：岗位能力模型沉淀、员工能力标签、胜任力差距分析、培养建议生成
3. **人才盘点**：9 宫格辅助、高潜人才识别建议、人才风险提示、后备梯队建议
4. **组织分析**：团队结构分析、管理跨度分析、核心岗位风险、人才密度洞察、关键能力缺口识别
5. **管理者支持**：团队绩效总结、晋升建议草稿、调岗建议说明、团队能力建设建议

**Agent 形态**：Performance Agent / Capability Agent / Talent Review Agent / Org Analyst Agent / Manager Support Agent

**北极星指标**：让人才与组织决策更清晰
- 衡量：绩效总结时间、人才盘点准备时间、组织分析产出质量

**预计周期**：4~6 个月

> ⚠️ V4 一定不要太早做——需要稳定的数据结构、更强的组织模型、更高信任度、更多客户真实数据反馈。

#### V5：HR OS / Multi-Agent Platform — 全 HR Agent 平台

**定位**：Seven — The AI operating system for HR work

把前面分散能力统一成真正的 HR OS，不是简单加功能。

**核心能力**：
1. **统一 Agent Hub**：所有 HR Agent 聚合为系统，用户不再关心在哪个模块，直接交给系统完成任务
2. **统一任务中心**：支持自然语言任务分发（"我想招一个 AI 产品经理" / "分析销售团队能力缺口"），系统自动分派给不同 Agent
3. **统一知识层**：汇总 JD/简历/员工档案/制度文档/绩效记录/面试反馈/团队结构数据/组织历史数据 → HR 专属知识图谱
4. **统一工作流编排**：招聘需求 → 筛选 → 面试 → 入职 → 试用期 → 绩效 → 晋升/调岗，Agent 之间接力而非模块割裂
5. **管理层与 HRBP 智能驾驶舱**：组织能力画像、人才风险预警、招聘效率趋势、绩效分布异常、人才梯队健康度

**北极星指标**：让 HR 团队拥有统一的 AI 工作操作层
- 衡量：Agent 覆盖率、任务自动完成率、HR 团队整体人效提升

**预计周期**：持续演进（不是短版本，而是平台化阶段）
