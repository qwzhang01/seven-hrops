# 平台内置工具集（Builtin ToolPacks）

> 版本：v2.1 | 日期：2026-06-01 | 状态：已实施（Phase G）
>
> 对应代码层：`L2 平台层` 与 `L1 内核层` 之间的"基础设施 SDK"
>
> 适用范围：**所有内置 Agent / 用户自定义 Agent / Marketplace Agent 都通过 ToolRegistry 引用本文档中的工具**

---

## 〇、概念定位（必读）

> ⚠️ **本文件曾用名"元能力底座"**，因为"元能力"一词与 HSAS 体系中的 `kind: Capability`（业务能力）相互混淆，且"元（meta）"暗示了"能力的能力"这种并不存在的层级关系，已废弃。
>
> 📖 **术语双轨**（中文"能力"何时用 / 英文 `Capability` 何时用 / 与"功能"/"技能"的边界）统一见 [14-通用约定.md](./14-通用约定.md) §〇。

### 0.1 本文档描述什么

本文档描述的是 **软件出厂时随包内置的、所有 Agent 都可调用的底座 Tool 集合**。它在 HSAS 规范中精确对应：

| HSAS 字段 | 取值 | 说明 |
|---|---|---|
| `kind` | `ToolPack`（HSAS-Spec §1.2 已预留） | 一组围绕同一底座功能打包的 Tool |
| `metadata.source` | `builtin` | 软件内置，**不是**用户后期通过 AgentBuilder/Marketplace 加进来的 |
| 加载时机 | 启动时随 `ToolRegistry.registerAll()` 一同注入 | 早于任何业务 Agent 加载 |
| 是否可卸载 | ❌ 不可卸载，仅可在设置中按 ToolPack 维度 disable | 与 builtin Capability 一致 |

> 一句话：**这是软件预置的底座 Tool，不是用户后期自己加进去的 Tool。** 用户/Marketplace 后续也可以发布同样 `kind: ToolPack` 的工具包，区别只在 `metadata.source = user | marketplace`，三者**同构**。

### 0.2 与 HSAS 四大概念的关系

```
Capability（kind: Capability）       ← 用户视角的入口卡片
   ↓ agentName 绑定
Agent（kind: Agent）                  ← 执行体（System Prompt + 工具白名单）
   ↓ spec.tools.allowed 白名单引用
Tool（ToolRegistry 中的具体函数）     ← 本文档的最小单位
   ↑ 归属于
ToolPack（kind: ToolPack）            ← 本文档的章节单位（10 个内置 ToolPack）
   ↑ source: builtin
平台内置（本文档范围）                 ← 与用户/Marketplace ToolPack 同构
```

### 0.3 与"业务能力（Capability）"的边界

| 维度 | 业务能力（Capability） | 内置工具集（Builtin ToolPack） |
|---|---|---|
| HSAS `kind` | `Capability` | `ToolPack` |
| 面向 | 用户（HR） | Agent / 平台 |
| UI 入口 | 左侧能力卡（组件名 `CapabilityCard`，渲染对象为 Capability） | 无 UI 入口；通过 ToolRegistry 暴露给 Agent |
| 例子 | 简历筛选、JD 优化、面试评价 | `fs-toolpack`、`docgen-toolpack`、`webserver-toolpack` |
| 是否可被用户卸载 | builtin 不可卸载（仅可 disable）；user/marketplace 可卸载 | builtin 不可卸载（仅可整包 disable） |
| `metadata.source` | builtin / user / marketplace | builtin（本文档专属）/ user / marketplace（在其他文档） |

## 一、内置 ToolPack 全景图

按"职责"分成 10 个 ToolPack，共 60+ 个 Tool。**每个 ToolPack 都是一个独立的 `kind: ToolPack` 注册单元**，可以独立 enable/disable，可以独立审计。

```

│                  source: builtin · 不可卸载 · 同构于用户 ToolPack       │
├────────────────────┬────────────────────┬───────────────────────────┤
│ 1. fs-toolpack      │ 2. docgen-toolpack  │ 3. parse-toolpack         │

│   媒体处理 & 转写    │   本地 HTTP 服务     │   邮件 / IM / 浏览器       │
├────────────────────┼────────────────────┼───────────────────────────┤
│ 7. scheduler-toolpack│ 8. system-toolpack  │ 9. security-toolpack     │
│   定时 / 事件        │   剪贴板 / 通知 / 打开│   PII / 合规 / 审计        │
├────────────────────┴────────────────────┼───────────────────────────┤
│       10. metrics-toolpack                                            │
│           Token / 调用计量 / 链路追踪                                  │

                                  ↑
                         统一通过 ToolRegistry 注册（kind: ToolPack）
```

每个 ToolPack 既包含**给 Agent 用的 MCP 工具**，又可附带**给前端用的 Service / Store / 组件**（前端组件不属于 Tool，但与 ToolPack 一同发布）。

## 二、ToolPack 1：`fs-toolpack`（工作空间 & 文件 IO）

> 来源：function.md「按聊天内容自动创建工作空间」「工作空间分一次任务/定时任务」

### 2.1 设计目标

- 每个能力会话 = 一个工作空间目录，命名规范统一（`{capabilityId}_{yyyyMMdd_HHmmss}`）
- 所有产物（输入/中间/报告）都在工作空间内，方便归档、分享、重跑
- 跨平台路径（macOS / Windows）由 Tauri 统一处理，业务 Agent 不用关心

### 2.2 Tool 清单

| 工具名 | 类别 | 描述 |
|---|---|---|
| `create_workspace_dir` | safe | 创建标准 5 层目录的工作空间，写入 `_meta.json` |
| `list_files` | safe | 列出工作空间任一子目录文件，支持扩展名过滤 |
| `read_file` / `write_file` | safe / write | 读写文本文件（限制在工作空间内） |
| `copy_files_to_workspace` | write | 把拖拽 / 系统路径的文件复制到工作空间子目录 |
| `move_file` / `delete_file` | write / sensitive | 工作空间内文件移动 / 删除（删除需确认） |
| `compute_file_hash` | safe | SHA256 哈希，用于去重（同一份简历不重复解析） |
| `archive_workspace` | write | 把工作空间打成 zip，便于分享 / 备份 |
| `restore_workspace` | write | 从 zip 还原工作空间 |

### 2.3 标准目录约定

```
~/SevenHROps/workspaces/{capabilityId}_{timestamp}/
├── 00_inputs/         📥 用户提供的输入（JD/简历/录音原始件）
├── 01_raw/            📄 抓取/收取下来的原始资源
├── 02_parsed/         🔍 解析后的结构化 / 文本快照
├── 03_intermediate/   📊 评分、打分等中间产物
├── 04_reports/        📝 最终报告（md / html / docx / pptx）
└── _meta.json         { capabilityId, createdAt, projectId, agent, tags[] }
```

> 详细的目录差异约定与混合存储策略见 [14-通用约定.md](./14-通用约定.md) §一。

### 2.4 为什么不做"全局文件系统"

用户自定义 Agent 默认**只能**访问 `当前工作空间` + `~/.seven-hrops/<agent>/data/`，避免误操作把简历泄露到桌面或下载文件夹。详见 [00-总架构设计.md](./00-总架构设计.md) 第十二章。

## 三、ToolPack 2：`docgen-toolpack`（文档生成 & 导出）

> 来源：function.md 反复出现的「导出 word/ppt/html」

### 3.1 设计目标

- **一份 Markdown 源 → 多端输出**：md / html / docx / pptx / pdf
- 提供一组"HR 默认主题 + 业务专题主题"模板，避免每个 Agent 自己写 CSS
- 大文档（>50 页）走流式生成 + 进度回调，前端可看到 ToolCall 进度

### 3.2 Tool 清单

| 工具名 | 类别 | 描述 |
|---|---|---|
| `render_markdown_to_html` | safe | Markdown → HTML（支持 mermaid / 数学公式 / 代码高亮） |
| `export_to_html` | write | HTML → 落地到 `04_reports/*.html`，可选嵌入资源 |
| `export_to_word` | write | Markdown / HTML → DOCX（pandoc 或 docx-templater） |
| `export_to_pptx` | write | Markdown → PPTX，按章节分页（python-pptx / officegen） |
| `export_to_pdf` | write | HTML → PDF（headless chromium，懒加载） |
| `apply_template` | safe | 给 Markdown 套主题（hr-default / screening / interview / report） |
| `merge_documents` | write | 多份 Markdown 合并 + 自动生成目录 |
| `compress_images_in_doc` | write | 文档内大图压缩，导出体积可控 |

### 3.3 主题（Template）系统

```
~/.seven-hrops/templates/
├── hr-default/         默认通用主题
│   ├── word.docx          DOCX 母版（页眉、字体、目录样式）
│   ├── slide.pptx         PPT 母版（封面、节、表格、图表）
│   └── html/
│       ├── theme.css
│       └── partials/   header / footer / toc
├── screening/          简历筛选专题（评分表配色、雷达图样式）
├── interview/          面试提纲 / 评价
└── report/             综合报告
```

业务 Agent 在 Manifest 中声明：

```yaml
exportProfile:
  template: screening
  formats: [docx, pdf, html]
```

### 3.4 与业务能力的解耦

简历筛选、面试评价、访谈提纲、撰写报告……都通过同一组 `export_to_*` Tool 落地，**不再每个能力实现自己的 PPT 生成器**。

## 四、ToolPack 3：`parse-toolpack`（文件解析 & 抽取）

> 来源：function.md「简历进入初筛」「邮件附件下载」「资料放到工作区」

### 4.1 Tool 清单

| 工具名 | 类别 | 描述 |
|---|---|---|
| `parse_pdf` | safe | PDF → 纯文本 + 结构（标题/表格/图） |
| `parse_docx` | safe | Word → 纯文本 + 段落结构 |
| `parse_excel` | safe | xlsx → JSON 表格 |
| `parse_image_ocr` | safe | 图片简历 / 截图 → 文字（Tesseract / 云 OCR） |
| `parse_resume` | safe | 简历专用结构化（姓名/教育/经历/技能/项目） |
| `parse_jd` | safe | JD 专用结构化（岗位/职责/要求/福利/公司信息） |
| `extract_email_attachments` | write | 从邮件提取附件并落到 `01_raw/` |
| `chunk_long_document` | safe | 长文档按语义切片，便于大模型逐段处理 |
| `dedupe_resumes` | safe | 通过哈希 + 关键字段去重 |

### 4.2 多版本解析策略

- **第一选择**：本地解析（pdfium / docx-rs），离线可用，零费用
- **第二选择**：云解析（针对扫描件 / 复杂排版），用户在设置中显式启用
- 解析后产物双写：DB（结构化字段） + 工作空间 `02_parsed/{id}.txt|json`，避免重复解析

## 六、ToolPack 5：`webserver-toolpack`（本地 HTTP 服务 / 局域网分享）

> 来源：function.md「内置 web server，渲染自己生成的静态 html，也可以改 / 内网访问，任何一个人都是一个服务器」

### 6.1 设计目标

- 一行调用就能把 `04_reports/report.html` 通过 `http://192.168.1.x:7301/r/xxx` 分享出去
- 支持「面试提纲在线展示」「笔试题在线作答并提交」两类典型场景
- 提供权限：临时链接（带 token，可过期）、内网开放、密码保护、点对点广播

### 6.2 Tool 清单

| 工具名 / 服务 | 类别 | 描述 |
|---|---|---|
| `webserver_start` | system | 启动本机 HTTP server，返回 `host:port` |
| `webserver_publish` | write | 发布一个目录 / 单个文件，返回访问 URL + token |
| `webserver_unpublish` | write | 取消发布 |
| `webserver_list` | safe | 列出当前所有发布项及访问统计 |
| `webserver_get_lan_ip` | safe | 获取本机局域网 IP（多网卡时让用户选择） |
| `webserver_create_form` | write | 发布一个支持表单提交（如笔试作答）的页面 |
| `webserver_collect_submission` | write | 收集表单提交结果，回写到工作空间 `03_intermediate/` |
| `webserver_qrcode` | safe | 生成访问二维码，方便手机端打开 |

### 6.3 典型流程：笔试在线作答

```
exam-designer Agent 生成笔试题 (Markdown)
        ↓
export_to_html(template='exam')  → 04_reports/exam.html
        ↓
webserver_publish('04_reports/exam.html', { mode: 'form', expireMinutes: 120 })
        ↓
返回：http://192.168.1.23:7301/r/abc123  + 二维码
        ↓
候选人在手机扫码作答 → webserver_collect_submission
        ↓
答卷写入 03_intermediate/submissions/{candidateId}.json
        ↓
通知 interview-evaluator Agent 自动评分
```

### 6.4 安全约束

- 默认仅监听 `0.0.0.0` 局域网，**不打洞、不上公网**
- 每个发布项都有独立 token，token 不出现在路径前缀，无法枚举
- 表单收集走「白名单字段」校验，避免被注入恶意 payload
- `webserver_*` 全部归属 `system` 类，**用户/Marketplace Agent 默认禁用**，需用户显式授权（见 §十二）

## 八、ToolPack 7：`scheduler-toolpack`（调度 / 定时 / 事件）

> 来源：function.md「工作空间分一次任务、定时任务性质」「设置自动收集简历的定时器」

### 8.1 Tool 清单

| 工具名 | 类别 | 描述 |
|---|---|---|
| `schedule_cron` | system | 注册 Cron 任务（每日 9 点跑一次简历抓取） |
| `schedule_once` | system | 一次性延时任务（30 分钟后提醒发面试反馈） |
| `schedule_event` | system | 事件订阅（新邮件 / IM 命令 / 文件变更触发） |
| `schedule_list` | safe | 列出当前所有定时任务 |
| `schedule_pause` / `schedule_resume` | write | 暂停 / 恢复 |
| `schedule_cancel` | write | 取消任务 |
| `task_history` | safe | 历史执行记录（成功 / 失败 / 耗时 / 输出摘要） |

### 8.2 与"工作空间一次任务/定时任务"的关系

工作空间在创建时声明 `kind`：

```json
{ "capabilityId": "resume-screening", "kind": "scheduled", "cron": "0 9 * * *" }
```

平台启动时会扫描所有 `kind=scheduled` 工作空间并注册定时器；任务执行结果落到工作空间 `04_reports/cron-runs/{date}.md`。

### 8.3 失败治理

- 定时任务失败自动重试 3 次，间隔指数退避
- 连续失败 3 天 → IM 通知用户 + 自动暂停（避免空转）
- 每次执行写入 `task_history`，UI 「能力管理」页可视化

## 十五、Manifest 中如何引用内置工具

业务 Agent 通过 `spec.tools.allowed` 按 Tool 名字逐个引用——**不直接引用 ToolPack 整包**，这样能精确控制权限。例如简历筛选 Agent：

```yaml
apiVersion: hsas.seven-hrops/v1
kind: Agent
metadata:
  name: screener
  source: builtin
spec:
  mode: subagent
  skills: [screener]
  tools:
    allowed:
      # —— 来自 fs-toolpack ——
      - create_workspace_dir
      - list_files
      - read_file
      - write_file
      - copy_files_to_workspace
      # —— 来自 parse-toolpack ——
      - parse_pdf
      - parse_docx
      - parse_resume
      - dedupe_resumes
      # —— 来自 docgen-toolpack ——
      - render_markdown_to_html
      - export_to_html
      - export_to_word
      - export_to_pptx
      - apply_template
      # —— 来自 webserver-toolpack（可选）——
      - webserver_publish
      - webserver_qrcode
      # —— 业务专属工具（在本能力的 src/capabilities/screening/tools.ts 中定义）——
      - parse_resume_batch
      - save_screening_result
      - generate_screening_report
```

**好处**：

1. 一眼看清"业务能力 = 内置 Tool 组合 + 极少业务专属工具"
2. 用户复制一份 Manifest 改名，就能造一个变种 Agent，内置部分零成本
3. ToolRegistry 校验时按 Tool 名查到 `defaultAllowedSources`，沙箱自动按 `metadata.source` 拒绝越权（详见 [11-HSAS-Spec.md](./11-HSAS-Spec.md) §九）

> **⚠️ 不要把 ToolPack 名（如 `fs-toolpack`）直接写进 `tools.allowed`**——白名单粒度是 Tool 而非 ToolPack。整包授权只在用户「设置 → ToolPack 启用开关」一级生效。

## 十七、设计决策

| 决策 | 选项 | 选择 | 理由 |
|---|---|---|---|
| 内置 Tool 暴露方式 | 直接 import / MCP 工具 | **MCP 工具 + ToolPack 注册** | 统一沙箱、统一审计、统一计量；与用户/Marketplace ToolPack 同构 |
| Word 导出实现 | pandoc 二进制 / docx-templater / officegen | **docx-templater + 模板** | 不依赖外部二进制，体积小 |
| PPT 导出实现 | python-pptx（子进程） / officegen / 自研 | **officegen（Node 原生）** | 无 Python 依赖，跨平台稳定 |
| PDF 导出实现 | wkhtmltopdf / headless chromium / @react-pdf | **headless chromium（懒加载）** | 与 HTML 主题完全一致，CSS 复用率 100% |
| 录音转写 | 云 ASR / whisper.cpp | **whisper.cpp 优先 + 云为补充** | 隐私优先；云需用户显式开启 |
| Web Server 框架 | Fastify / Hono / Tauri 内置 | **Hono on Tauri** | 包体小、API 现代、内嵌方便 |
| Cron 实现 | node-cron / 操作系统级 | **node-cron + 持久化到 SQLite** | 不依赖系统 launchd / 任务计划 |
| IM SDK | 三家各自 SDK / 统一 webhook | **统一 webhook 抽象 + 各家 adapter** | 业务 Agent 只看到统一接口 |
| 浏览器自动化 | Puppeteer / Playwright / Browser-Use | **Playwright + Browser-Use 协议** | 跨浏览器、社区活跃、有 LLM 协议 |
| 脱敏触发点 | 上行 LLM / 下行存盘 / 仅日志 | **上行 LLM + 审计日志** | 隐私优先，存盘保留原始（HR 工作需要） |
