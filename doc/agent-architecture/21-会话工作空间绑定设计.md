## 1. 设计目标与动机

### 问题

在 `session-workspace-binding` 之前，会话（Session）和工作空间（Workspace）是完全独立的两个概念：

- 用户创建会话后，需要手动选择或创建工作空间
- 切换会话时，文件树区域不会自动切换，显示的仍是上一个会话的文件
- 没有机制区分"需要文件输入输出的会话"和"纯对话会话"

### 目标

1. **会话与工作空间自动绑定**：创建会话时，根据 Capability 的 `needsWorkspace` 字段自动创建/跳过 Workspace
2. **切换会话时文件树同步切换**：`setActiveSession` 触发 `workspaceStore` 的联动
3. **无 Workspace 的会话显示友好提示**：文件树区域显示空状态，而非空白

## 2. `needsWorkspace` 字段

### 定义

在 `src/platform/hsas/schema.ts` 的 `CapabilitySpec` 中新增：

```typescript
needsWorkspace?: boolean  // 默认 false
```

### 语义

| 值 | 含义 |
|---|---|
| `true` | 该能力需要文件输入输出（如简历筛选、JD 优化、报告生成） |
| `false` | 该能力是纯对话（如 assistant、music-radio） |
| 未声明 | **Lint 错误**（`NEEDS_WORKSPACE_NOT_DECLARED`），必须显式声明 |

### 各 Capability 的声明

| Capability | `needsWorkspace` |
|---|---|
| `resume-screening` | `true` |
| `jd-optimization` | `true` |
| `report-writing` | `true` |
| `interview-eval` | `true` |
| `interview-outline` | `true` |
| `written-test` | `true` |
| `assistant` | `false` |
| `employee-interview` | `false` |
| `orchestrator` | `false` |
| `music-radio` | `false` |

## 3. Session 与 Workspace 的绑定契约

### 绑定时机

```
createSession(capabilityId)
  ├── resolveWorkspaceNeeds(capabilityId) → true
  │   ├── workspaceStore.createWorkspace(capabilityId)
  │   └── workspaceStore.bindSession(sessionId, workspaceId)
  └── resolveWorkspaceNeeds(capabilityId) → false
      └── (不创建 Workspace，session.workspaceId = undefined)
```

### 切换时机

```
setActiveSession(sessionId)
  ├── sessionWorkspaceMap[sessionId] 存在
  │   └── workspaceStore.switchToSession(sessionId)
  │       ├── set currentWorkspaceId = workspaceId
  │       ├── set currentWorkspacePath = workspace.path
  │       └── refreshFileTree(workspaceId)
  └── sessionWorkspaceMap[sessionId] 不存在
      └── workspaceStore.clearActive()
          ├── set currentWorkspaceId = null
          ├── set currentWorkspacePath = null
          └── set fileTree = []
```

### 解绑时机

```
deleteSession(sessionId)
  └── workspaceStore.unbindSession(sessionId)
      └── delete sessionWorkspaceMap[sessionId]
```

### 数据结构

```typescript
// workspaceStore 新增字段
sessionWorkspaceMap: Record<string, string>  // sessionId → workspaceId

// Session 类型新增字段（db.ts）
workspace_id?: string  // 可选，向后兼容

// SessionInfo 新增字段（session.ts）
workspaceId?: string  // 可选，向后兼容
```

## 4. UI 联动流程

```
用户点击会话列表中的某个会话
  └── SessionList → sessionStore.setActiveSession(id)
        ├── set activeSessionId = id
        ├── ChatPanel 监听 activeSessionId 变化
        │   ├── clearAIMessages()
        │   ├── loadMessages(id)
        │   ├── ChatInput key={activeSessionId} → React 重新挂载，清空输入框
        │   └── switchLock = true (100ms) → 防止误发消息
        └── workspaceStore.switchToSession(id) 或 clearActive()
              └── ResourcePanel 监听 currentWorkspaceId 变化
                    ├── currentWorkspaceId 存在 → 显示文件树
                    └── currentWorkspaceId 为 null → 显示空状态
```

### 空状态 UI

当 `currentWorkspaceId` 为 null 时，`ResourcePanel` 显示：

```
📁 当前会话无需工作空间
   这是一个纯对话会话
   [上传文件]  ← disabled，v2.1+ 实现临时 Workspace 创建
```

---

## 5. 向后兼容性

- `workspaceId` / `workspace_id` 是**可选字段**，旧数据不受影响
- 旧 Session（无 `workspace_id`）在 `sessionWorkspaceMap` 中没有条目，`setActiveSession` 时会调用 `clearActive()`，显示空状态
- `sessionWorkspaceMap` 是**内存中的**，重启后重新绑定（`arch-session-db-persistence` 会解决持久化问题）
