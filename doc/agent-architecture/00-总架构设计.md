# Seven HROps — Agent 总架构设计

> 版本：v2.0 | 日期：2026-05-27 | 状态：设计稿（平台化升级版）

---

## 〇、平台定位（v2.0 关键升级）

> **Seven HROps 不是"9 个 HR 能力（Capability）的集合"，而是一个"Agent / Skill 运行平台"，9 个能力是平台上自带的"种子应用"。**
>
> 📖 全局术语规范（窄义"能力" / 广义"功能" / `CapabilityCard` / `Capability` 双轨命名等）见 [14-通用约定.md](./14-通用约定.md) §〇。

平台分三层：

```
┌──────────────────────────────────────────────────────────┐
│  L3 应用层（Capabilities，可插拔）                        │
│  ─ builtin: 简历筛选 / JD优化 / 面试提纲 / 笔试 / 面试评价│
│             访谈提纲 / 撰写报告 / 音乐电台 / 协调者        │
│  ─ user:    用户自然语言描述 → Agent Builder 自动生成     │
│  ─ market:  从远程仓库 / GitHub 安装                     │
└──────────────────────────────────────────────────────────┘
                           ↑
┌──────────────────────────────────────────────────────────┐
│  L2 平台层（Platform Runtime）                            │
│  ─ CapabilityRegistry：能力注册中心（builtin + user）     │
│  ─ AgentLoader / SkillLoader：从配置/文件/DB 动态加载     │
│  ─ ToolRegistry：工具注册中心 + 分级授权                  │
│  ─ AgentBuilder：自然语言 → Agent 配置（元 Agent）        │
│  ─ Sandbox：用户自定义 Agent 的权限隔离                   │
└──────────────────────────────────────────────────────────┘
                           ↑
┌──────────────────────────────────────────────────────────┐
│  L1 内核层（Agent Runtime Core，已有）                    │
│  ─ Agent.Service / Skill.Service / Permission.Service     │
│  ─ Session / SessionProcessor / Tool 执行器               │
└──────────────────────────────────────────────────────────┘
```

**核心原则：**
1. **内置即插件**：9 个内置能力的加载方式与用户自定义能力**完全相同**，没有特权
2. **配置即代码**：每个 Agent / Skill / Capability 都是一份**清单文件（Manifest）**，启动时加载，热插拔
3. **声明式优先**：用户通过自然语言或表单描述需求，平台生成清单文件，**不需要写代码**
4. **安全可信**：用户自定义 Agent 默认进入"低权限沙箱"，敏感工具需显式授权

## 一、背景与目标

### 1.1 当前系统问题

- `ChatWorkspace.handleSend` 走本地正则匹配（`recognizeIntent`），没有接入 Agent Runtime
- 音乐电台通过关键词硬编码触发，无情绪感知功能
- HR 能力各自独立，没有全局任务协调者
- **Agent / Skill 全部硬编码在 `BUILT_IN_AGENTS` / `BUILT_IN_SKILLS` 字面量里，无法扩展**
- **能力（Capability）和 Agent 是 1:1 写死的字符串映射，新增能力必须改源码重新打包**
- **没有"用户自定义"的概念，更没有"自然语言生成 Agent"的入口**

### 1.2 v2.0 目标

> **核心原则（必读）**：v2.0 不是"先粗糙跑起来，将来再重构成平台"，而是 **"用平台方案的最小子集先跑一个能力，再批量铺开剩余能力"**。
> 因此下文的"内置能力层"和"平台底座层"不是两套并行方案，而是 **同一套架构的两个层次**：底座先建（Phase 0），内置能力在底座上长出来（Phase 1～4），AgentBuilder 与沙箱在能力齐全后启用（Phase 5）。
> **任何内置能力都不允许绕过底座（Loader / Registry / ToolRegistry）走"临时硬编码"通路**，这是从设计期就要守住的边界，否则后期迁移成本巨大。

**A. 平台底座层（Phase 0，先于一切能力开发）**：
- **HSAS 规范（HROps Agent & Skill Spec）**：统一 Agent / Skill / Capability 的清单格式
- **CapabilityRegistry**：能力注册中心，内置能力 seed 注入，用户自定义走 SQLite 持久化
- **AgentLoader / SkillLoader**：把 Manifest 注入 `Agent.Service` 的唯一渠道
- **ToolRegistry + 分级授权**：MCP 工具按敏感度分级、登记元数据（拦截逻辑可在 Phase 5 启用）
- **Sandbox 接口预留**：Phase 0 仅保留 hooks，对内置 Agent 走"无沙箱"通路，Phase 5 再启用

**B. 内置能力层（Phase 1～4，9 个能力跑在底座上）**：
- 所有对话全部走 Agent Runtime，彻底移除 `intentService` 硬编码逻辑
- 每个能力对应一个专属 Agent，**通过 YAML Manifest + Loader 加载**，差异仅在于 System Prompt 和工具列表
- Phase 1 先把"简历筛选"一个能力完整跑通（Manifest → Loader → Registry → UI → Tool 调用），作为底座的试金石；验证通过后再批量推进剩余 8 个能力
- 第 9 个能力 **全局协调者（Orchestrator）** 负责任务状态感知、跨能力调度、企微桥接
- 音乐电台升级为情绪感知型 DJ Agent

**C. 开放生态层（Phase 5，平台能力齐全后启用）**：
- **AgentBuilder**：用户用自然语言描述需求 → 元 Agent（agent-builder）自动生成 Manifest → 写入注册中心
- **Sandbox 启用**：用户/marketplace 来源的 Agent 进入沙箱，敏感工具需显式授权
- **MarketPlace（v2.1+）**：从 Git / 远程仓库安装第三方 Agent / Skill / ToolPack 包

> 说明：内置 Tool 集合（10 个 builtin ToolPack）属于 **A. 平台底座层** 的一部分，与底座同步交付，不是独立的

## 三、HSAS 规范（HROps Agent & Skill Spec）★ v2.0 核心

> 把"Agent / Skill / Capability"标准化为三类清单文件，**内置和用户自定义共用同一套规范**。

### 3.1 Agent Manifest

YAML 格式（也支持 JSON / 直接存 SQLite）：

```yaml
# ~/.seven-hrops/agents/my-tech-screener.yaml
apiVersion: hsas.seven-hrops/v1
kind: Agent
metadata:
  name: my-tech-screener           # 唯一 ID（仅小写字母-数字-连字符）
  displayName: 互联网技术岗筛选     # UI 显示名
  description: 针对互联网研发岗位的简历筛选 Agent
  source: user                     # builtin | user | marketplace
  version: 1.0.0
  author: zhangsan
  icon: 🧑‍💻
  createdAt: 2026-05-27T10:00:00Z
spec:
  mode: subagent                   # primary | subagent
  basePrompt: |
    你是专注于互联网研发岗（前端/后端/算法）的简历筛选助手。
    评分维度调整为：
    - 技术栈匹配（占 40%）
    - 项目复杂度（占 25%）
    - 大厂/独角兽经历（占 15%）
    - 学历（占 10%）
    - 文化契合（占 10%）
  contextTemplate: |
    当前工作空间：{{workspacePath}}
    当前 JD：{{jdContent}}
    当前项目：{{projectId}}
  skills:                          # 引用的 Skill 列表
    - screener
    - tech-stack-detector          # 用户自定义 Skill
  tools:                           # 工具白名单
    allowed:
      - list_files
      - parse_resume_batch
      - save_screening_result
      - generate_screening_report
    deny:
      - delete_resume              # 显式禁止
  permission:                      # 沿用现有 Permission.Rule 格式
    - { permission: read,      pattern: "*", action: allow }
    - { permission: screening, pattern: "*", action: allow }
    - { permission: write,     pattern: "screening_results/*", action: allow }
  model:
    provider: anthropic            # 可选，覆盖默认
    modelID: claude-4.7-sonnet
    temperature: 0.3
  capabilityBinding:               # 可选：直接绑定为一个 UI 能力
    capabilityId: tech-screening
    displayName: 技术岗简历筛选
    icon: 🧑‍💻
    category: hr-screening
```

### 3.3 Capability Manifest

```yaml
apiVersion: hsas.seven-hrops/v1
kind: Capability
metadata:
  name: tech-screening
  displayName: 技术岗简历筛选
  description: 互联网研发岗的简历筛选能力
  source: user
  icon: 🧑‍💻
  category: hr-screening           # UI 分组
  order: 100                       # 排序权重
spec:
  agentName: my-tech-screener      # 绑定哪个 Agent
  contextKeys:                     # 需要 contextBuilder 注入的 key
    - workspacePath
    - jdId
    - projectId
  entryPrompt: |                   # 用户进入能力时的开场白
    我是技术岗简历筛选助手，请提供 JD 和简历文件夹路径。
  inputSchema:                     # 可选：固定参数表单
    - { key: experienceYears, label: 年限, type: number }
```

### 3.4 三种清单的关系

```
Capability ──绑定──→ Agent ──引用──→ Skill
    ↑                  ↑                ↑
 用户在UI看到的能力   实际执行体     可复用的知识/流程
```

- **Capability** 是 UI 入口（左侧 CapabilityCard）
- **Agent** 是执行体，定义身份、工具、权限
- **Skill** 是知识/流程的封装，可被多个 Agent 复用

## 四、平台层设计（L2 Platform Runtime）

### 4.1 CapabilityRegistry — 能力注册中心

```typescript
// src/platform/capabilityRegistry.ts
export interface CapabilityRecord {
  manifest: CapabilityManifest
  source: 'builtin' | 'user' | 'marketplace'
  enabled: boolean
  installedAt: number
}

export class CapabilityRegistry {
  // 启动时调用：注入 9 个内置能力（不写 DB，每次启动 seed）
  async loadBuiltins(): Promise<void>

  // 启动时调用：从 SQLite 读取用户能力 + 文件系统扫描
  async loadUserCapabilities(): Promise<void>

  // 用户在 UI 新建能力
  async install(manifest: CapabilityManifest, source: 'user'): Promise<void>

  // 卸载（仅 user/marketplace 可卸载，builtin 只能 disable）
  async uninstall(name: string): Promise<void>

  enable(name: string): void
  disable(name: string): void

  list(filter?: { source?: string; enabled?: boolean }): CapabilityRecord[]
  get(name: string): CapabilityRecord | undefined
}
```

### 4.2 AgentLoader / SkillLoader — 清单加载器

```typescript
// src/platform/agentLoader.ts
export class AgentLoader {
  constructor(private agentService: AgentService, private toolRegistry: ToolRegistry) {}

  // 从 YAML / JSON / DB 行加载，校验后注册到 Agent.Service
  async load(manifest: AgentManifest): Promise<void> {
    manifestValidator.validate(manifest)            // schema 校验
    this.toolRegistry.assertAllowed(manifest)       // 工具白名单校验（含权限分级）
    const info = manifestToAgentInfo(manifest)      // 转换为 Agent.Info
    await this.agentService.register(info)
  }

  async unload(name: string): Promise<void> {
    await this.agentService.unregister(name)        // ★ 需要给 Agent.Service 增加 unregister
  }

  async reload(manifest: AgentManifest): Promise<void> {
    await this.unload(manifest.metadata.name)
    await this.load(manifest)
  }
}
```

### 4.3 ToolRegistry — 工具注册中心 + 分级授权

```typescript
// src/platform/toolRegistry.ts
export type ToolCategory = 'safe' | 'write' | 'sensitive' | 'system' | 'builder'

export interface ToolMeta {
  name: string
  category: ToolCategory
  riskLevel: 'low' | 'medium' | 'high' | 'critical'
  description: string
  // 默认对哪些来源的 Agent 开放
  defaultAllowedSources: Array<'builtin' | 'user' | 'marketplace'>
  // 是否需要每次调用前 ask 用户
  requireApproval: boolean
}

export const TOOL_REGISTRY: Record<string, ToolMeta> = {
  list_projects:        { category: 'safe',      riskLevel: 'low',    defaultAllowedSources: ['builtin','user','marketplace'], requireApproval: false, /*...*/ },
  save_jd:              { category: 'write',     riskLevel: 'medium', defaultAllowedSources: ['builtin','user','marketplace'], requireApproval: false, /*...*/ },
  delete_resume:        { category: 'sensitive', riskLevel: 'high',   defaultAllowedSources: ['builtin'],                      requireApproval: true,  /*...*/ },
  send_wecom_message:   { category: 'sensitive', riskLevel: 'high',   defaultAllowedSources: ['builtin'],                      requireApproval: true,  /*...*/ },
  update_task_status:   { category: 'system',    riskLevel: 'medium', defaultAllowedSources: ['builtin'],                      requireApproval: false, /*...*/ },
  create_agent_manifest:{ category: 'builder',   riskLevel: 'medium', defaultAllowedSources: ['builtin'],                      requireApproval: true,  /*...*/ },
  // ...
}
```

效果：
- 用户自定义 Agent 默认**不能**调用 `delete_resume` / `send_wecom_message`
- 在 UI 上启用敏感工具时，弹窗提示"该工具会发送外部消息，是否授权？"
- 内置 Agent 不受限（保持现有行为）

### 4.4 Sandbox — 用户 Agent 沙箱

| 维度 | builtin | user | marketplace |
|---|---|---|---|
| 默认工具范围 | 全部 | safe + write | safe |
| 文件读写路径 | 任意 | 仅工作空间 + `~/.seven-hrops/` | 仅 Skill 包内部 |
| 网络请求 | 任意 | 仅白名单域名 | 显式声明 + 用户审核 |
| Token 配额（每会话） | 无限 | 100K | 50K |
| 调用频率 | 无限 | 60次/分钟 | 30次/分钟 |

实现要点：在 `SessionProcessor` 调用 Tool 之前插入一个 `sandboxGuard` 钩子。

## 五、AgentBuilder — 自然语言生成 Agent ★

> **这是 v2.0 的旗舰特性**：用户用一句话描述需求，平台自动生成完整的 Agent + Capability + Skill 清单。

### 5.1 用户体验

```
[用户在主界面点 "+" → 新建能力]
         ↓
弹出 AgentBuilder 对话框
         ↓
用户输入："我要做一个面向校招生的简历筛选 Agent，
          重点看实习经历、ACM/Kaggle 比赛、开源项目，
          淡化工作年限要求"
         ↓
agent-builder Agent 启动（这是一个内置的元 Agent）
         ↓
1) 分析需求 → 生成 Capability + Agent Manifest
2) 推荐已有的 Skill（screener / compliance）
3) 询问：是否需要新建一个"实习经历评估" Skill？
         ↓
用户确认 → 写入 SQLite + 调用 AgentLoader.load()
         ↓
左侧 CapabilityCard 立即出现"校招筛选"卡片，点击就能用
```

### 5.2 AgentBuilder Agent 配置

```typescript
'agent-builder': {
  name: 'agent-builder',
  mode: 'primary',
  description: '元 Agent：把自然语言需求转换为合法的 Agent / Skill / Capability Manifest',
  prompt: `你是 Seven HROps 平台的 Agent 工厂。

你的工作流程：
1. 与用户对话，澄清三件事：
   - 这个 Agent 要解决什么 HR 问题？
   - 输入是什么（JD / 简历 / 录音 / 文档）？
   - 输出是什么（评分表 / Markdown 报告 / Word / 推送通知）？
2. 调用 list_available_tools 查看可用工具
3. 调用 list_available_skills 查看可复用 Skill
4. 生成三份清单（Capability / Agent / 必要时新建 Skill）
5. 调用 validate_manifest 验证合法性
6. 展示给用户预览，确认后调用 install_capability 落地

关键规则：
- 优先复用已有 Skill，只在没有合适的时新建
- 工具白名单必须最小化（仅必需）
- 用户自定义 Agent 默认 source: user（沙箱）
- prompt 要写得具体、可执行，避免空话`,
  permission: [{ permission: '*', pattern: '*', action: 'allow' }],
  tools: [
    'list_available_tools',
    'list_available_skills',
    'list_available_agents',
    'create_agent_manifest',     // 写入 agent_manifests 表
    'create_skill_manifest',     // 写入 skill_manifests 表 / 文件系统
    'create_capability_manifest',// 写入 capabilities 表
    'validate_manifest',
    'install_capability',        // 调用 CapabilityRegistry.install + AgentLoader.load
  ],
  temperature: 0.5,
}
```

### 5.3 配套 MCP 工具

| 工具 | 作用 |
|---|---|
| `list_available_tools` | 返回 ToolRegistry 中所有工具及其元数据 |
| `list_available_skills` | 返回所有已注册 Skill 的元数据 |
| `list_available_agents` | 返回所有已注册 Agent 的元数据 |
| `create_agent_manifest` | 生成 Agent 清单（JSON）并校验 |
| `create_skill_manifest` | 生成 Skill 清单 + 在 `~/.seven-hrops/skills/` 创建目录 |
| `create_capability_manifest` | 生成 Capability 清单 |
| `validate_manifest` | 调 manifestValidator 做合法性校验 |

### 5.4 安全闭环

- AgentBuilder 生成的 Manifest 默认 `source: user`

- 如果用户想给自定义 Agent 加敏感工具，必须**手动**在能力管理页打勾授权

## 六、内置能力（9 个）的加载方式

> **关键原则：内置能力走和用户自定义能力**完全相同**的加载流程，只是数据来源不同。**

```typescript
// src/platform/builtinSeed.ts
import screenerManifest from './manifests/screener.yaml'
import jdOptimizerManifest from './manifests/jd-optimizer.yaml'
// ... 其他 7 个

export const BUILTIN_AGENT_MANIFESTS: AgentManifest[] = [
  screenerManifest,
  jdOptimizerManifest,
  // ...
]

export const BUILTIN_CAPABILITY_MANIFESTS: CapabilityManifest[] = [...]
export const BUILTIN_SKILL_MANIFESTS: SkillManifest[] = [...]
```

启动流程：

```typescript
// src/platform/bootstrap.ts
export async function bootstrapPlatform() {
  // 1. 注册所有工具到 ToolRegistry
  toolRegistry.registerAll(TOOL_REGISTRY)

  // 2. 加载内置 Skill / Agent / Capability（走 Loader，不走特殊路径）
  for (const m of BUILTIN_SKILL_MANIFESTS)      await skillLoader.load(m)
  for (const m of BUILTIN_AGENT_MANIFESTS)      await agentLoader.load(m)
  for (const m of BUILTIN_CAPABILITY_MANIFESTS) await capabilityRegistry.install(m, 'builtin')

  // 3. 加载用户能力（SQLite + 文件系统扫描）
  await capabilityRegistry.loadUserCapabilities()

  // 4. UI 订阅 capabilityRegistry 变更，刷新 CapabilityCard 列表
}
```

**这意味着**：
- 用户写出来的 `my-tech-screener` 和内置的 `screener` 在运行时**没有任何差别**，只是 `source` 字段不同
- 卸载内置能力是不允许的，但可以 `disable`（左侧不显示卡片）
- 用户可以**继承**内置能力：把 `screener` 的 Manifest 复制一份，改名 `my-screener`，调整 prompt → 就有了私人定制版

## 十二、工作空间约定（What & Why）★

> **What**：所有能力在创建/进入时都自动建立一个工作空间用于管理中间产物，路径统一在 `~/SevenHROps/workspaces/`，命名规则 `{capabilityId}_{yyyyMMdd_HHmmss}`。
>
> **Why**：① 给用户一个稳定可见的"这次任务的所有材料都在这里"的物理目录，方便归档和分享；② 跨设备/重启后能基于元信息重建状态；③ 让评分、报告、导出文件等中间产物有标准化的落点，不污染用户其他目录。
>
> **核心原则**：**结构化字段进 DB（用于查询/排序/聚合），大文本与原始文件进工作空间（用于查看/归档/分享）**。
>
> 📖 详细 How（路径解析规范、5 个标准子目录的语义、各能力的子目录映射、混合存储策略表、双通道文件录入、右侧资源面板组件契约）见 **[14-通用约定.md](./14-通用约定.md) §一**。

## 十三、能力静默切换（What & Why）★

> **What**：用户不必去左侧点击能力卡，只要在通用对话框表达意图（"帮我筛简历"、"这帮 JD 优化下"、"打开音乐电台"等），`assistant` 自动路由到对应能力，过程中**不说多余的"我要切换到 X"**，左侧卡片直接高亮+右侧工作空间切换+由子 Agent 接手回答。
>
> **Why**：① 降低能力发现门槛——很多用户不会读说明书，自然语言是最低成本入口；② 保持对话连贯——切换动作不打断用户思路；③ 可撤销——顶部 Toast「已进入《简历筛选》· 点这里切回」给用户损退机会，不强制锁定。
>
> **关键机制**：`assistant` Agent 拥有 `activate_capability(capabilityId, reason, confidence)` 工具（ToolRegistry `system` 分级），前端在 `agentService` 监听 `tool-start: activate_capability` 事件触发 UI 切换；置信度 ≥ 0.7 才切换，否则提问确认。
>
> 📖 详细 How（体验准则、`activate_capability` 工具签名、前端 `agentService` 监听伪代码、`assistant` Manifest 完整片段、9 个能力的触发关键词速查表）见 **[14-通用约定.md](./14-通用约定.md) §二**。

## 十四、权限分级与沙箱（v2.0 新增）

### 12.1 工具分级

| 类别 | 示例 | builtin | user | marketplace |
|---|---|---|---|---|
| safe | list_*, get_* | ✅ | ✅ | ✅ |
| write | save_*, write_file | ✅ | ✅ | ⚠️ 默认禁用，需用户授权 |
| sensitive | delete_*, send_wecom_message, transcribe_audio | ✅ | ⚠️ 需授权 | ❌ 禁止 |
| system | update_task_status, get_user_emotion_history | ✅ | ❌ | ❌ |
| builder | create_agent_manifest, install_capability | ✅（仅 agent-builder） | ❌ | ❌ |

### 12.2 沙箱限制

- **文件系统**：用户 Agent 只能读写 `当前工作空间` 和 `~/.seven-hrops/<agent-name>/data/`
- **网络**：用户 Agent 默认禁止 fetch；marketplace 包必须在 Manifest 中声明 `network.allowedHosts`
- **Token 配额**：用户 Agent 单会话默认 100K Token，超限自动暂停并询问用户
- **频率限制**：用户 Agent 60 次/分钟工具调用，超限节流

### 12.3 Manifest 签名（v2.1+）

Marketplace 安装的 Skill 包必须有作者签名，平台校验通过才能加载（防止恶意 Skill）。

## 十六、关键设计决策

| 决策 | 选项 | 选择 | 理由 |
|---|---|---|---|
| Manifest 格式 | JSON / YAML / TOML | **YAML（主），JSON（API/DB 存储）** | YAML 可读性强，适合用户编辑；JSON 适合机器交换 |
| 内置能力存储 | 字面量 / YAML 文件 / DB seed | **YAML 文件，编译时打包** | 与用户能力同构，且不依赖运行时网络 |
| 用户能力存储 | YAML 文件 / SQLite | **SQLite + 可选导出 YAML** | 跨平台一致、事务安全；YAML 用于分享/备份 |
| Skill 目录约定 | OpenCode 风格 / Anthropic 风格 | **Anthropic 风格（SKILL.md + frontmatter）** | 与生态对齐，未来可直接复用第三方 Skill |
| Tool 注册方式 | 装饰器 / 配置表 | **配置表（TOOL_REGISTRY）** | 集中管理元数据，便于权限审计 |
| 元 Agent 是否内置 | 内置 / 用户自建 | **内置（agent-builder）** | 保证生成质量、安全性；用户不需要"先学会写 Agent 才能写 Agent" |
| 沙箱实现层 | OS 级 / 应用级 | **应用级（在 SessionProcessor / FS 命令中加 guard）** | Tauri 桌面应用够用，避免引入 docker/wasm 复杂度 |
