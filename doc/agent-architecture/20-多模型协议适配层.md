# 多模型协议适配层（Multi-Model Protocol Adapter）

> 本文档是 [runtime-multimodel-protocol-adapter](../../openspec/changes/runtime-multimodel-protocol-adapter/) OpenSpec change 落地后的稳定参考。
>
> - 目标读者：需要新增 / 切换 LLM provider 的开发者；想理解豆包等"prompt-style"模型工具调用的运维同学。
> - 单一可信源：本文档解释 Why & What；How 全部对应到 [`src/agent-runtime/provider/`](../../src/agent-runtime/provider/) 的代码。

---

## 〇、为什么要有这一层

### 0.1 多模型适配的两个老问题

| 问题 | 老代码症状 | 后果 |
|---|---|---|
| **新增一个 provider 要侵入 `createAdapter` switch case** | 4 个 case：`openai` / `anthropic` / `ollama` / `default`。新增 19 个国内外主流 provider 等于改 19 次主路径 | 主路径越改越胖，PR review 困难 |
| **豆包等模型把 tool-call 编码进文本流，AI SDK 直接当普通文本透传** | UI 显示一串乱码 `<\|FunctionCallBegin\|>{"name":"..."}<\|FunctionCallEnd\|>`，工具不被调用 | 火山方舟豆包模型卡完全无法使用，业务受阻 |

### 0.2 借鉴 opencode 的双层抽象

参考 [`opencode/packages/core/src/plugin/provider/`](../../.ext/opencode/packages/core/src/plugin/provider/) 与 [`opencode/packages/llm/src/`](../../.ext/opencode/packages/llm/) 的设计，本仓采用**双层抽象 + 装饰器**模型：

```
┌──────────────────────────────────────────────────────────┐
│  ProviderRegistry.resolve(providerID, config, modelID)   │← 单一入口
└──────────────────────────────────────────────────────────┘
                ▼ 路由到三类分支之一
┌─────────────────┬────────────────┬─────────────────────┐
│ ProviderPlugin  │ Dynamic Plugin │ OpenAI-Compat 兜底  │
│ （19 个具名）   │ （白名单）     │ （未知 + 显式协议） │
└─────────────────┴────────────────┴─────────────────────┘
                ▼ 拿到 sdkInstance + selectAdapter()
┌──────────────────────────────────────────────────────────┐
│           ProtocolAdapter（3 种协议族）                  │
│  ・openai-native       （含 reasoning-delta 的官方 GPT） │
│  ・openai-compatible   （绝大多数厂商）                  │
│  ・anthropic-messages  （Claude 块式 SSE，已被 SDK 归一）│
└──────────────────────────────────────────────────────────┘
                ▼ 可选包裹
┌──────────────────────────────────────────────────────────┐
│   PromptStyleToolCallDecorator（豆包 / Qwen 治本层）     │
│   把文本流中的 token 还原为 tool-call 事件               │
└──────────────────────────────────────────────────────────┘
                ▼
        Stream<LLMStreamEvent, Error> → ToolRuntime
```

## 一、三层抽象详解

### 1.1 ProviderPlugin —— 一个 provider = 一个文件

每个内置 provider 是 [`src/agent-runtime/provider/plugins/<id>.ts`](../../src/agent-runtime/provider/plugins/) 下一个 15–40 行的薄壳：

```ts
// plugins/volcengine.ts —— 火山方舟豆包
export const volcenginePlugin: ProviderPlugin = {
  id: "volcengine",
  defaultProtocol: "openai-compatible",
  defaultBaseURL: "https://ark.cn-beijing.volces.com/api/v3",
  promptStyle: "doubao", // ← 治本：默认开 prompt-style 装饰器
  createSdk: (cfg) =>
    createOpenAICompatible({
      name: "volcengine",
      apiKey: cfg.apiKey,
      baseURL: cfg.baseURL ?? "https://ark.cn-beijing.volces.com/api/v3",
    }),
};
```

新增主流 provider 只需新增一个文件 + barrel 注册一行。**不要改主路径**。

### 1.2 ProtocolAdapter —— 把 AI SDK 的 TextStreamPart 翻译成 LLMStreamEvent

实现位置：[`src/agent-runtime/provider/protocols/`](../../src/agent-runtime/provider/protocols/)

| 协议 ID | 来源场景 | 是否含 reasoning-delta |
|---|---|---|
| `openai-native` | `@ai-sdk/openai` 官方 GPT-4o / o1（含原生推理流） | ✅ |
| `openai-compatible` | 绝大多数厂商（千问 / 豆包 / DeepSeek / GLM / xAI / Groq / ...） | ❌ |
| `anthropic-messages` | `@ai-sdk/anthropic` Claude（SDK 已把 content_block_delta 归一为 TextStreamPart，协议层薄壳即可） | ❌ |

三个 adapter 在相同输入下产出**完全一致**的 `LLMStreamEvent` 序列（**同构契约**），由 [`anthropic-messages.test.ts`](../../src/agent-runtime/provider/protocols/anthropic-messages.test.ts) `it.each` 锁死。

### 1.3 PromptStyleToolCallDecorator —— 治本豆包

[`protocols/prompt-style-tool-call.ts`](../../src/agent-runtime/provider/protocols/prompt-style-tool-call.ts) 是装饰器：

- 包裹 inner ProtocolAdapter，**只对 `text-delta` 做 token 扫描**，其它事件原样透传
- 用 `Stream.mapAccum` 维护状态机 `idle | in-call`
- 跨 chunk 切割（极端到 1 字节 1 chunk）也能正确识别
- 未闭合 + abort 时 emit `LLMStreamEvent.error`，不静默丢失
- JSON parse 失败时降级 emit error，不 crash

支持的两种 prompt 风格（[`TOKEN_PATTERNS`](../../src/agent-runtime/provider/protocols/prompt-style-tool-call.ts) 常量）：

| ID | begin | end | 适用模型 |
|---|---|---|---|
| `"doubao"` | `<\|FunctionCallBegin\|>` | `<\|FunctionCallEnd\|>` | 火山方舟全系豆包模型 ([官方文档](https://www.volcengine.com/docs/82379/1330310)) |
| `"qwen"` | `<tool_call>` | `</tool_call>` | Qwen3-Coder 等显式编码工具调用的模型卡 ([官方文档](https://qwen.readthedocs.io/)) |

> **未来扩展**：阿里千问、月之暗面如果某天发布需要 prompt-style 编码的新模型卡，**只改对应 plugin 的 `promptStyle` 字段**即可，前端 / 业务层零改动。

## 二、安全约束：DynamicProvider 白名单 + 二次确认

[`plugins/dynamic.ts`](../../src/agent-runtime/provider/plugins/dynamic.ts) 允许用户在不修改代码的前提下接入未列举的 provider，但有三道闸门：

1. **白名单正则** `^@ai-sdk/[a-z0-9-]+$`：只允许 ai-sdk 官方命名空间下的小写包，拒绝任意 npm 包
2. **首次加载二次确认**：通过 [`@tauri-apps/plugin-dialog`](../../src-tauri/) 弹原生 confirm（消息体明确写出 npm 包名 + 警示），用户确认后写入 `<workspace>/.config/dynamic-providers-trusted.json`
3. **工厂形态校验**：动态 import 的模块必须暴露 `createXxx` 工厂，且工厂返回值必须含 `{ specificationVersion, provider, modelId }` 三字段

任意一道失败 fail-fast，对应 6 个 TaggedError：`UnknownProviderProtocolError` / `DynamicProviderNotAllowedError` / `DynamicProviderMissingPackageError` / `DynamicProviderUserDeclinedError` / `DynamicProviderNoFactoryError` / `DynamicProviderInvalidFactoryError`。

> **架构决策**（详见 [design.md Decision 9](../../openspec/changes/runtime-multimodel-protocol-adapter/design.md)）：trust list 的文件 IO 直接调 `@tauri-apps/plugin-fs` 而不走 `toolRegistry`——因为 toolRegistry 本身依赖 ProviderRegistry 解析 LLM，而 LLM 解析依赖 trust list，存在循环初始化风险。

## 三、配置范例

### 3.1 火山方舟豆包（最小工作配置）

[`src/agent-runtime-config.json`](../../src/agent-runtime-config.json) 的 `provider` 段：

```jsonc
{
  "provider": {
    "volcengine": {
      "apiKey": "<your-ark-api-key>",
      "protocol": "openai-compatible",
      "baseURL": "https://ark.cn-beijing.volces.com/api/v3",
      "promptStyle": "doubao"
    }
  },
  "model": {
    "default": {
      "providerID": "volcengine",
      "modelID": "doubao-1-5-pro-32k-250115"
    }
  }
}
```

> `protocol` / `promptStyle` / `baseURL` **都可省略**——`volcenginePlugin` 自带默认。但显式写出有两个收益：① 用户复制配置时一目了然 ② 日后 plugin 默认值变更不会无声生效。

### 3.2 Anthropic Claude（治国际主流）

```jsonc
{
  "provider": {
    "anthropic": {
      "apiKey": "<your-anthropic-key>",
      "protocol": "anthropic-messages",
      "baseURL": "https://api.anthropic.com/v1"
    }
  },
  "model": {
    "default": {
      "providerID": "anthropic",
      "modelID": "claude-sonnet-4-20250514"
    }
  }
}
```

### 3.3 Dynamic 模式（未列举的厂商）

以接入 Cerebras 为例（**前提**：`@ai-sdk/cerebras` 在 npm 上确实存在）：

```jsonc
{
  "provider": {
    "dynamic": {
      "apiKey": "<your-cerebras-key>",
      "protocol": "openai-compatible",
      "dynamicPackage": "@ai-sdk/cerebras"
    }
  },
  "model": {
    "default": {
      "providerID": "dynamic",
      "modelID": "llama-4-scout-17b-16e-instruct"
    }
  }
}
```

**首次加载行为**：
1. 前端 `LLMConfigModal` 客户端正则校验 `^@ai-sdk/[a-z0-9-]+$` 通过
2. 后端 `ProviderRegistry.resolve` 路由到 `resolveDynamicProvider`
3. Tauri 弹出原生 confirm 对话框（标题"动态 Provider 加载"，警告级别）
4. 用户点"信任并加载" → 写入 `~/<workspace>/.config/dynamic-providers-trusted.json` → 反射 `createCerebras` 工厂
5. 第二次启动应用直接放行（trust list 命中）

### 3.4 国产兼容族通用模板

DeepSeek / 智谱 GLM / Moonshot / 零一万物 / 腾讯混元都遵循同一形态——只换 `id` + `apiKey` + `baseURL`：

```jsonc
{
  "provider": {
    "deepseek": { "apiKey": "<key>", "baseURL": "https://api.deepseek.com/v1" },
    "zhipu":    { "apiKey": "<key>", "baseURL": "https://open.bigmodel.cn/api/paas/v4" },
    "moonshot": { "apiKey": "<key>", "baseURL": "https://api.moonshot.cn/v1" },
    "lingyiwanwu": { "apiKey": "<key>", "baseURL": "https://api.lingyiwanwu.com/v1" },
    "hunyuan":  { "apiKey": "<key>", "baseURL": "https://api.hunyuan.cloud.tencent.com/v1" }
  }
}
```

完整 19 + 1 项的覆盖矩阵见 [21-provider-coverage-matrix.md](./21-provider-coverage-matrix.md)。

## 四、扩展指南

### 4.1 我想新增一个内置 provider

1. 在 [`plugins/`](../../src/agent-runtime/provider/plugins/) 下新建 `<id>.ts`，仿照 [`alibaba.ts`](../../src/agent-runtime/provider/plugins/alibaba.ts) 写 15–40 行薄壳
2. 在 [`plugins/index.ts`](../../src/agent-runtime/provider/plugins/index.ts) 的 `BUILT_IN_PROVIDER_IDS` 数组追加 ID + `ProviderPlugins` 数组追加 plugin（顺序对齐由 `_pluginShape.test.ts` 守卫）
3. 在 [`LLMConfigModal.tsx`](../../src/components/modals/LLMConfigModal.tsx) 的 `PROVIDER_DISPLAY` 添加显示名 + emoji（**typecheck 会强制要求**：缺一项编译就过不去）
4. 跑 `pnpm exec tsc --noEmit` + `vitest run src/agent-runtime/provider/plugins`

### 4.2 我想新增一种 prompt 风格

1. 在 [`prompt-style-tool-call.ts`](../../src/agent-runtime/provider/protocols/prompt-style-tool-call.ts) 的 `PromptStyleID` 联合追加新字面（如 `"glm"`）
2. 在 `TOKEN_PATTERNS` 常量中新增对应 `{ begin, end }`
3. 在对应 plugin 的 `promptStyle` 字段挂上新风格
4. 在 [`prompt-style-tool-call.test.ts`](../../src/agent-runtime/provider/protocols/prompt-style-tool-call.test.ts) 加 Case F 覆盖跨 chunk + abort + JSON 失败三个场景
5. 在 [`LLMConfigModal.tsx`](../../src/components/modals/LLMConfigModal.tsx) 的 promptStyle 下拉选项追加一项
6. **真机 fixture 锚点（强制）**：见 4.2.1

### 4.2.1 真机 fixture 制度（prompt-style 强制）

> 制度来源：[`runtime-multimodel-real-machine-verification`](../../openspec/changes/runtime-multimodel-real-machine-verification/specs/platform-foundation/spec.md) §「ADDED Requirements」；同步纪律：[`frontend-conventions.md` §3.2.1](../../.codebuddy/rules/frontend-conventions.md)

**为什么需要这条制度**：`runtime-multimodel-protocol-adapter` change 落地后首次接入豆包真机即暴露——`<|FunctionCallBegin|>` token 之间 JSON 体在豆包真机是 `[{"name":"X","parameters":{...}}]`（顶层数组 + `parameters` 字段），而原 change 的 17 条契约/集成测试 fixture 全部凭对 OpenAI 习惯写成 `{"name":"X","arguments":{...}}`。CI 全绿但真机失败——因为**整套测试 fixture 没有任何真机锚点**。

**实操五步法**：

1. **抓包**：在 dev 模式下用 DevTools 打开 Network → 触发一次工具调用 → 找到 `chat/completions` 请求 → 复制 EventStream 中含 `<|FunctionCallBegin|>...<|FunctionCallEnd|>` 那一段 raw bytes。
2. **脱敏**：把 apiKey、Authorization、用户 ID 等敏感字段从字串中删除（仅保留与协议解析相关的部分）。
3. **入仓**：在 [`src/agent-runtime/provider/__fixtures__/`](../../src/agent-runtime/provider/__fixtures__/) 下创建 `<provider>-real.ts`，导出 const 字符串；docstring 必须包含：① 抓包时间 + 模型版本（或官方文档 URL + 章节锚点），② 字节级复刻警示，③ 敏感字段脱敏声明。参考实现：[`doubao-real.ts`](../../src/agent-runtime/provider/__fixtures__/doubao-real.ts)。
4. **加 case**：在 `integration-<provider>.test.ts` 追加一条 case，描述含 `(real-machine fixture)` 字样以便 grep 审计；fixture 直接 `import` 步骤 3 的 const 而不是就地 hardcode。
5. **守卫断言**：case 内除了断言「工具被以正确入参调用」外，SHOULD 加一组 fixture 字串自检（`expect(payload).toContain("parameters")` 之类），守住「fixture 文件不被悄悄改写为想当然形态」。

**没有真机环境时的兜底**：在该 change 的 design.md `Open Questions` 显式列出「JSON schema 形态稳定性」为待回放项，把对应 case 标记 `it.skip("real-machine fixture pending: <link>")`，等真机回放后再启用。**禁止跳过该制度**——结构性问题不能靠"反正现在跑得过 CI"自我催眠。

### 4.3 我想新增一种协议

> **谨慎**：这是侵入性最强的扩展，需要先在 OpenSpec 立项。

1. 在 [`_types.ts`](../../src/agent-runtime/provider/_types.ts) 的 `ProtocolID` 联合追加新字面
