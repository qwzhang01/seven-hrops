# 能力 01 — 简历筛选（resume-screening）

> 对应 Agent：`screener` | 模式：subagent | 优先级：P0
>
> 📌 **v2.0 平台化说明**：本文档保留旧版

## 一、能力描述

用户提供 JD 和简历文件夹，Agent 自动批量读取简历、逐一解析、按 JD 要求评分，最终生成筛选报告。

---

## 二、完整对话流程

### 2.1 入口（双通道）

- **通道 A — 显式点选**：用户点击左侧能力卡「简历筛选」→ 直接激活 `screener`
- **通道 B — 自然语言识别**：用户在通用对话框输入「帮我筛简历 / 帮我看看这批简历哪个合适」等 → `assistant`（primary Agent）识别意图后调用 `activate_capability('resume-screening')` → 前端**静默切换**到该能力（左侧卡片高亮、右侧资源面板切到对应工作空间），对话无缝继续，不打断用户思路

### 2.2 工作空间准备（自动）

用户进入能力后，若 `workspaceStore.currentWorkspace` 为空：

1. Agent 调用 `create_workspace_dir(capabilityId='resume-screening')`
2. Tauri 在 `~/SevenHROps/workspaces/` 下创建 `resume-screening_20260527_105930/` 目录（命名规则：`{capabilityId}_{yyyyMMdd_HHmmss}`，跨平台路径由 Tauri `dirs::home_dir()` 解析，macOS/Windows 自动适配）
3. 同步初始化 5 个子目录：`00_inputs/`、`01_resumes_raw/`、`02_resumes_parsed/`、`03_screening/`、`04_reports/`
4. 写入 `_meta.json`（capabilityId、createdAt、关联的 projectId）
5. 工作空间路径回写 `workspaceStore` + 数据库 `workspaces` 表

### 2.3 简历来源（双通道）

- **通道 A — 已在工作空间**：用户提前把简历放进 `01_resumes_raw/` → Agent 调用 `list_files` 直接列举
- **通道 B — 拖拽上传**：用户把简历文件拖到对话框 → 前端监听 `drop` 事件 → 调用 `copy_files_to_workspace(filePaths[], '01_resumes_raw')` → 文件复制到工作空间统一管理（避免后续路径丢失） → Agent 收到一条系统消息 `[已新增 N 份简历到工作空间]` 后继续流程

### 2.4 主对话流程

```
用户：帮我筛选简历
  ↓
screener Agent：请问 JD 是什么？（如果工作空间 00_inputs/jd.md 已存在则跳过）
  ↓
用户：JD 在工作空间里 / 粘贴 JD 文本（粘贴时自动写入 00_inputs/jd.md）
  ↓
screener Agent：调用 list_files('01_resumes_raw') 获取简历列表
  ↓
screener Agent：调用 parse_resume_batch(filePaths[]) 批量解析
   → 结构化字段写 DB resumes 表
   → 纯文本快照写 02_resumes_parsed/{resumeId}.txt
  ↓
screener Agent：逐一评分（5个维度），调用 save_screening_result 存库
  ↓
screener Agent：调用 generate_screening_report(projectId) 生成报告
   → 核心 Markdown 写 DB reports 表
   → 同步落一份到 04_reports/report.md
  ↓
流式输出报告内容到对话
  ↓
screener Agent：「报告已生成。需要导出成什么格式？
  [1] HTML（适合在线分享）
  [2] Word（适合打印归档）
  [3] PPT（适合汇报演示，Phase 3 提供）
  [4] 暂不导出」
  ↓
根据用户回复调用 export_to_html / export_to_word / export_to_pptx，导出文件落 04_reports/
```

## 七、本能力的差异说明

> 通用部分（工作空间路径/命名/双通道录入/混合存储原则/资源面板/共用工具）见 **[14-通用约定.md](./14-通用约定.md)**。本节仅列出简历筛选的特殊点。

### 7.1 子目录映射

| 标准位 | 本能力具体名 | 内容 |
|--------|-------------|------|
| `01_*_raw/` | `01_resumes_raw/` | 原始简历文件（PDF/Word） |
| `02_*_parsed/` | `02_resumes_parsed/` | 解析后的纯文本 + JSON 镜像（用于 RAG/重评） |
| `03_*/` | `03_screening/` | 评分中间产物 / DB 评分快照 |
| `04_reports/` | `04_reports/` | 筛选报告（MD 主体 + HTML/Word/PPTX 导出） |

### 7.2 本能力专属的存储落点

| 内容 | 存储位置 | 说明 |
|------|---------|------|
| 5 维度评分结果（技能/经验/学历/行业/文化） | **仅** DB `screening_results` 表 | 需要排序+筛选+跨简历对比，单一可信源避免一致性问题 |
| JD 原文 | 工作空间 `00_inputs/jd.md` + DB `jds` 表 | 双向同步：用户可直接编辑文件，也可在 UI 编辑 |
| 简历结构化字段 | DB `resumes` 表（主） + 工作空间 `02_resumes_parsed/{id}.json` 镜像 | 评分时频繁读 DB；JSON 镜像便于人工核对 |

### 7.3 报告导出格式选项

报告生成后，screener Agent 会主动询问用户导出格式（详见 §2.4 主对话流程末尾）：

| 选项 | 工具 | 阶段 |
|------|------|------|
| HTML（适合在线分享） | `export_to_html` | Phase 2 |
| Word（适合打印归档） | `export_to_word` | Phase 2 |
| PPT（适合汇报演示） | `export_to_pptx` 配 `screening` 模板 | **Phase 3** |
| 暂不导出 | — | — |

### A.4 用户继承示例 — `my-tech-screener`

如果用户希望"在内置筛选基础上更看重开源贡献"，可以让 [AgentBuilder](./10-AgentBuilder.md) 帮其生成下面这份继承型 Manifest，无需改源码：

```yaml
apiVersion: hsas.seven-hrops/v1
kind: Agent
metadata:
  name: my-tech-screener
  displayName: 互联网技术岗筛选
  description: 在内置 screener 基础上增加开源贡献维度
  source: user
  version: 1.0.0
  author: zhangsan
  icon: 🧑‍💻
  createdAt: 2026-05-27T10:00:00Z
spec:
  mode: subagent
  inheritFrom:
    name: screener
    overrides: [basePrompt]
  basePrompt: |
    你是专注于互联网研发岗的简历筛选助手。
    评分维度调整为：技能(35%) + 经验(20%) + 项目复杂度(20%) + 开源贡献(15%) + 文化契合(10%)
    其余规则继承自内置 screener。
  tools:
    allowed:
      - list_files
      - parse_resume_batch
      - save_screening_result
      - generate_screening_report
  capabilityBinding:
    capabilityId: tech-screening
    autoCreate: true
```

> 这份用户 Manifest 的安装流程见 [10-AgentBuilder.md](./10-AgentBuilder.md)；其默认沙箱配置（FS / 网络 / 工具白名单）见 [13-沙箱实现细节.md](./13-沙箱实现细节.md)。
