# 测试策略 — 测试金字塔与覆盖率契约

> **本文档目标**：统一全项目的测试理念、工具选型、覆盖率目标与 PR 准入标准，避免散落在 [10] §八 / [13] §十二 / [15] §十二 中的零散要求互相打架。
>
> **强制等级**：
> - 🔴 **强制**：未达标的代码不能合并到主干
> - 🟡 **推荐**：建议做但不阻塞合并
> - 🟢 **可选**：视团队精力推进

---

## 〇、设计原则

1. **测试金字塔，不是冰淇淋筒**：单测 ≫ 集成 ≫ E2E。E2E 慢且脆弱，只覆盖**最关键**的端到端路径。
2. **安全相关代码无妥协**：沙箱 / 校验器 / 权限拦截**必须**有单测，覆盖率 ≥ 90%。
3. **业务能力文档自带场景表**：[01-简历筛选.md](./01-简历筛选.md) 等已有"5 种典型场景"清单——这些直接转成集成测试用例即可。
4. **测试是文档**：测试名用"应该 / should"句式，能从测试名直接看懂业务规则。

---

## 一、测试金字塔（本项目目标比例）

```
       ╱ ╲       E2E (Playwright)
      ╱ 5 ╲      ─ 仅 3~5 条主链路，Phase 1 后再做
     ╱─────╲
    ╱  20   ╲    集成测试 (Vitest + tauri-mock)
   ╱─────────╲   ─ 跨模块端到端，重点：bootstrap / 工具调用全链路 / 沙箱拦截
  ╱    75     ╲
 ╱─────────────╲ 单元测试 (Vitest / cargo test)
                  ─ 纯函数 / Schema 校验 / Loader / 各 ToolPack
```

数字是"覆盖代码量"的近似比例，不是测试用例数。

## 二、工具选型

| 层 | 语言 | 工具 | 配置 |
|---|---|---|---|
| 前端单测 | TypeScript | **Vitest** + `@vitest/coverage-v8` | `vitest.config.ts` |
| 前端 Mock | TypeScript | `vi.mock` + `msw`（HTTP 拦截） | — |
| Tauri 命令 Mock | TypeScript | `@tauri-apps/api/mocks`（官方） | `mockIPC` |
| Effect Schema 测试 | TypeScript | Vitest + `@effect/schema/Schema` | 直接用 `Schema.decodeSync` |
| Rust 单测 | Rust | `cargo test` | 内联 `#[cfg(test)]` |
| 集成测试 | TypeScript | Vitest（开 happy-dom 环境） | `vitest.integration.config.ts` |
| E2E | TypeScript | **Playwright** + Tauri WebDriver（Phase 1 后） | — |

> **不要引入 Jest / Mocha / Chai**：项目已用 Effect 生态，Vitest 兼容性最好且更快。

## 八、Mock 策略

### 8.1 LLM 调用

**绝不**在测试中真实请求 LLM。统一用 `vi.mock` 替换 `Agent.Service.chat`：

```ts
vi.mock('../agent-runtime/agent', () => ({
  Agent: {
    Service: {
      chat: vi.fn().mockResolvedValue({ content: 'mocked', toolCalls: [] }),
    },
  },
}))
```

### 8.2 Tauri 命令

用官方 `@tauri-apps/api/mocks`：

```ts
import { mockIPC } from '@tauri-apps/api/mocks'

beforeEach(() => {
  mockIPC((cmd, args) => {
    if (cmd === 'load_user_capabilities') return mockCapabilities
    if (cmd === 'write_text_file') return null
    throw new Error(`Unmocked command: ${cmd}`)
  })
})
```

### 8.3 SQLite

集成测试中用 in-memory SQLite：

```ts
import Database from 'better-sqlite3'
const db = new Database(':memory:')
db.exec(fs.readFileSync('src-tauri/migrations/v2_capabilities.sql', 'utf8'))
```

> Rust 单测同理：`rusqlite::Connection::open_in_memory()`。

### 8.4 时间

用 `vi.setSystemTime` 控制 `Date.now()`，确保 `installed_at` / `created_at` 可断言。

## 九、什么时候**不**写测试？🟢

承认现实：不是所有代码都值得测。**可以不写**测试的情况：

- UI 组件的纯展示部分（建议改用 Storybook 视觉回归）
- 一次性脚本（`scripts/seed-data.ts` 这类）
- 显然的 getter / setter
- 类型定义文件（`types.ts`）

**但绝不能跳过测试**的情况（即便很急）：
- 任何带 `Validator` / `Guard` / `Permission` 字样的模块
- 任何写 SQLite 的事务（数据丢失零容忍）
- 任何 Rust 端处理路径或 host 的代码
