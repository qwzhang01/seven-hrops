## 〇、术语与命名公约（v2.0 新增 · 全局可信源）

> **目的**：消除"能力""Skill""Capability"在不同文档中的歧义，让所有文档作者、UI 文案、代码命名都用同一套词汇。
>
> 本节是全局唯一规则，其他文档（00 ~ 13）必须与本节一致；不一致时以本节为准。

### 0.1 四个核心概念

| 中文（用户向 / UI 文案） | 英文（开发者向 / 代码标识） | HSAS `kind` | 含义 |
|---|---|---|---|
| **能力**（首次出现注 *Capability*） | **Capability** | `Capability` | 用户在左侧能力卡上看到、点击进入的 UI 入口。绑定一个 Agent + 一组 contextKeys + 入口提示词。**例**：简历筛选、JD 优化、音乐电台。|
| **Agent** | **Agent** | `Agent` | 真正的执行体——System Prompt + 工具白名单 + 权限规则。一个 Capability 必须绑定一个 Agent；一个 Agent 可被多个 Capability 复用。 |
| **技能**（首次出现注 *Skill*） | **Skill** | `Skill` | 可被多个 Agent 复用的"知识/流程包"（SKILL.md + 资源目录），按 Anthropic Skill 生态约定。**例**：tech-stack-detector、compliance-check。 |
| **工具集 / 工具包**（首次出现注 *ToolPack*） | **ToolPack** / **Tool** | `ToolPack` | 平台或第三方提供的底座工具集合。`Tool` 是最小调用单位（如 `list_files`），`ToolPack` 是一组同主题 Tool 的注册单元（如 `fs-toolpack`）。**例**：见 [12-平台内置工具集.md](./12-平台内置工具集.md)。 |

### 0.2 易混淆词的规范用法

| 词语 | 必须替换为 | 备注 |
|---|---|---|
| ~~"元能力"~~ | `ToolPack`（开发者向）/ "内置工具集"（用户向） | 已废弃。原因：与 `Capability` 字面冲突，且"元（meta）"暗示了不存在的层级关系。 |
| ~~"CapabilityCard"~~ | `CapabilityCard` | 组件名修正——它渲染的对象是 Capability，不是 Skill。 |
| 广义"能力"（指代功能/特性） | "功能" | 例：~~"无情绪感知能力"~~ → "无情绪感知**功能**"；~~"运行时注销能力"~~ → "运行时注销**功能**"；~~"桌面级能力"~~ → "桌面级**功能**"。 |
| 窄义"能力"（指代 `kind: Capability` 实体） | 视语境分轨（见 §0.3） | — |

### 0.3 双轨命名规则（窄义"能力" / Capability）

**开发者向语境** —— 一律用 `Capability`：
- 架构图 / 数据流图的实体框
- HSAS Manifest 字段名 / 注释
- 类名 / 接口名 / 文件名（`CapabilityRegistry`、`capabilityId`、`capability-installed` 事件）
- ToolRegistry 元数据描述
- 错误码 / 校验信息

**用户向语境** —— 保留中文"能力"，**首次出现**括注 `（Capability）`：
- UI 按钮 / 菜单 / Toast / Modal 文案（"能力管理"、"新建能力"、"已安装能力列表"）
- 业务能力文档标题（[07-撰写报告.md](./07-撰写报告.md) 中"能力 07 — 撰写报告"等保留）
- 用户引导词、对话提示词

> **判断口径**：如果这段文字最终是 HR 用户读到的，用"能力"；如果只是开发者读到的，用 `Capability`。

### 0.4 标识符大小写公约

| 场景 | 写法 | 示例 |
|---|---|---|
| TypeScript / Manifest 字段 | `camelCase` | `capabilityId`、`agentName`、`displayName` |
| MCP 工具入参 schema | `snake_case` | `capability_id`、`agent_name` |
| 文件名 / 目录名 | `kebab-case` | `agent-builder.yaml`、`my-tech-screener/` |
| HSAS `metadata.name` | `kebab-case`，正则见 [11-HSAS-Spec.md](./11-HSAS-Spec.md) §1.3 | `resume-screening`、`tech-stack-detector` |
| 事件名（EventEmitter / IPC） | `kebab-case` | `capability-installed`、`tool-start` |
| SQLite 表名 / 列名 | `snake_case` | `agent_manifests`、`installed_at` |

> **特别说明（字段名 vs 字段值）**：上表约束的是"**字段名 / 变量名 / 文件名**"的大小写。"**标识符的取值**"（如 `capabilityId` 这个字段的具体值 `"resume-screening"`）始终遵循各自实体的命名正则——见 [11-HSAS-Spec.md](./11-HSAS-Spec.md) §1.3。例：JS 代码里写 `capability.capabilityId === 'resume-screening'`，左边字段名用 camelCase，右边值用 kebab-case，两条规则不冲突。

## 一、工作空间约定

### 1.1 路径与命名规则

| 项 | 约定 |
|----|------|
| **根目录** | `~/SevenHROps/workspaces/`（跨平台统一） |
| **macOS / Linux 实际路径** | `/Users/<user>/SevenHROps/workspaces/` |
| **Windows 实际路径** | `C:\Users\<user>\SevenHROps\workspaces\` |
| **路径解析方式** | Tauri 端使用 `dirs::home_dir()`，**禁止硬编码路径** |
| **工作空间命名** | `{capabilityId}_{yyyyMMdd_HHmmss}` |
| **命名示例** | `resume-screening_20260527_105930`、`interview-eval_20260527_140212` |
| **创建时机** | 用户进入某能力但 `workspaceStore.currentWorkspace` 为空时，由 Agent 调用 `create_workspace_dir` 自动建立 |
| **元信息回写** | 创建后同步写入 `_meta.json` + 数据库 `workspaces` 表 |

### 1.2 标准目录结构

> **设计目标**：5 个一级子目录覆盖"输入 → 中间产物 → 最终交付物"的完整生命周期，让用户在右侧资源面板能一眼看懂用途。

```
~/SevenHROps/workspaces/{capabilityId}_{timestamp}/
├── 00_inputs/                   📥 用户输入材料（JD、参考文档、提示词等）
├── 01_*_raw/                    📄 原始上传文件（按业务实体命名子目录）
├── 02_*_parsed/                 🔍 解析后的可读快照（纯文本 + JSON 镜像）
├── 03_*/                        📊 业务中间产物（评分、转写、清洗结果等）
├── 04_reports/                  📝 最终报告（MD 主体 + HTML/Word/PPTX 导出）
└── _meta.json                   📌 工作空间元信息
                                    {
                                      "capabilityId": "resume-screening",
                                      "createdAt": "2026-05-27T10:59:30+08:00",
                                      "projectId": "proj_xxx",
                                      "version": "1.0"
                                    }
```

### 1.3 子目录命名差异约定

`01_` `02_` `03_` 三个中间目录的具体命名由各能力按业务实体替换 `*`，请保持语义清晰：

| 能力 | 01_*_raw | 02_*_parsed | 03_* |
|------|----------|-------------|------|
| 简历筛选 | `01_resumes_raw/` | `02_resumes_parsed/` | `03_screening/` |
| 面试提纲 | `01_jd_refs/` | `02_resume_refs/` | `03_outline_drafts/` |
| 面试评价 | `01_audio_raw/` | `02_transcripts/` | `03_evaluation/` |
| 访谈提纲 | `01_briefs/` | — | `03_outline_drafts/` |
| 撰写报告 | `01_materials/` | `02_excerpts/` | `03_drafts/` |
| 笔试准备 | `01_jd_refs/` | — | `03_question_bank/` |
| JD 优化 | `01_jd_raw/` | — | `03_jd_drafts/` |
| 音乐电台 | （不创建工作空间，纯流式能力） | | |

> 各能力只需在自己的文档里写一行"本能力子目录映射"即可，不必重复贴整棵树。

### 1.4 混合存储策略

> **核心原则**：**结构化字段进 DB（用于查询/排序/聚合），大文本与原始文件进工作空间（用于查看/归档/分享）。**

| 内容类型 | DB | 文件系统 | 说明 |
|----------|------|----------|------|
| 原始上传文件（PDF/Word/音频） | ❌ | ✅ `01_*_raw/` | 体积大、二进制，文件系统原生适配；右侧面板可直接打开 |
| 解析后的结构化字段 | ✅ 主 | ❌ | 需查询/排序/跨实体关联；评分/检索时频繁读取 |
| 解析后的纯文本快照 | ❌ | ✅ `02_*_parsed/{id}.txt` | 用于 RAG / 重新评分，避免重复解析 PDF（耗时） |
| 解析后的 JSON 镜像 | ❌ | ✅ `02_*_parsed/{id}.json` | DB 字段的人类可读副本，便于人工核对 |
| 评分 / 评价 / 推荐结果 | ✅ 主 | ❌ | 结构化、需排序+对比，单一可信源避免一致性问题 |
| 报告 Markdown 主体 | ✅ 主 | ✅ 镜像 `04_reports/report.md` | DB 为可信源，文件为用户可见镜像 |
| 导出文件（HTML/Word/PPTX） | ❌ | ✅ `04_reports/` | 用户交付物，文件系统直接可分享 |
| 工作空间元信息 | ✅ `workspaces` 表 | ✅ `_meta.json` | 双向同步，便于跨设备迷失后重建 |

### 1.5 双通道文件录入

| 通道 | 入口 | 处理逻辑 |
|------|------|---------|
| **A. 工作空间已有** | 用户预先把文件拷到 `01_*_raw/` | Agent 直接调 `list_files` 列举 |
| **B. 拖拽上传** | 用户把文件拖到 `ChatWorkspace` 对话框 | 前端监听 `drop` 事件 → 调用 `copy_files_to_workspace(filePaths[], '01_*_raw')` → 复制到工作空间 → 返回路径数组交付 Agent → Agent 收到系统消息 `[已新增 N 份文件到工作空间]` 后继续流程 |

### 1.6 右侧资源面板（`WorkspaceFileTree.tsx`）

- 监听 `workspaceStore.currentWorkspace`，递归调用 `list_files` 展示文件树
- 4 个一级目录使用固定 emoji 图标（📥 / 📄 / 🔍 / 📊 / 📝），让用户一眼看懂用途
- 双击文件 → PDF/Word/HTML 用系统默认应用打开；MD/JSON/TXT 在右侧内嵌预览
- Tauri `notify` watcher 监听文件变更，实时刷新树
- 顶部固定操作条：「打开工作空间」「在 Finder/资源管理器中显示」「清理旧工作空间」

## 二、能力静默切换

> 用户不必去左侧点击能力卡（`CapabilityCard`）。只要在通用对话框表达意图（"帮我筛简历"、"这帮 JD 优化下"、"打开音乐电台"等），`assistant` 会自动路由到对应能力（Capability）。

### 2.1 体验准则

| 准则 | 说明 |
|------|------|
| **静默切换** | 识别成功后**不要**说多余的"我要帮你切换到 X"，直接：高亮左侧能力卡 → 切换右侧工作空间 → 由子 Agent 接手回答 |
| **对话连贯** | 不创建新会话，复用 `capabilitySession.getOrCreate(capabilityId)` 后把原始请求透传给子 Agent |
| **可撤销** | 顶部出现轻量 Toast：「已进入《简历筛选》能力 · 点这里切回」，给用户损退机会 |
| **置信度阈值** | 仅当 `assistant` 评估意图置信度 ≥ 0.7 才切换；否则提问确认「你是想做简历筛选吗？」 |
| **识别不准不乱用** | 含糊表达（"帮我看看"）默认不切换；只有出现能力关键词（"筛简历"、"评估候选人"、"出题"等）才路由 |

### 2.2 实现机制

#### 2.2.1 `activate_capability` 工具

- 注册到 ToolRegistry `system` 分级，**仅 `assistant` Agent 可用**
- 工具签名：

```typescript
server.tool(
  "activate_capability",
  "Silently switch to a specified capability and delegate the original user request to its sub-agent",
  {
    capability_id: z.string().describe("Target capability ID, e.g. 'resume-screening'"),
    reason: z.string().describe("Why this capability matches the user intent (for audit log)"),
    confidence: z.number().describe("Intent recognition confidence, 0~1"),
  },
  async ({ capability_id, reason, confidence }) => {
    // 仅校验 capability_id 是否合法、是否已启用；真正的副作用由前端监听 tool-start 事件触发
    return { ok: true, capability_id, confidence }
  }
)
```

#### 2.2.2 前端在 `agentService` 监听 `tool-start: activate_capability`

```typescript
// 伪代码
agentService.on('tool-start', (evt) => {
  if (evt.tool !== 'activate_capability') return
  const { capability_id } = evt.input
  capabilityRegistry.assertEnabled(capability_id)         // 校验能力可用
  workspaceStore.switchTo(capability_id)                   // 需时自动建工作空间
  capabilitySession.getOrCreate(capability_id)             // 复用或新建会话
  uiStore.highlightCapabilityCard(capability_id)           // 左侧卡片高亮
  uiStore.showToast(`已进入《${displayName}》· 点这里切回`)
})
```

#### 2.2.3 `assistant` 在同一轮调用 `delegate_to_subagent`

`activate_capability` 执行成功后，`assistant` 立刻调用 `delegate_to_subagent(capability_name, originalUserMessage)`，让对应子 Agent 接管输出，整个过程对用户无缝。

### 2.3 `assistant` Manifest 关键片段

```yaml
spec:
  basePrompt: |
    你是 Seven HROps 的默认入口助手。职责：
    1. 识别用户意图是否属于 9 个内置能力之一（简历筛选 / JD 优化 / 面试提纲 / 笔试准备 / 面试评价 / 访谈提纲 / 撰写报告 / 音乐电台 / 全局协调者）
    2. 若是且置信度 ≥ 0.7 → 调用 activate_capability 并 delegate_to_subagent
    3. 若不是 → 直接以通用助手身份回复
    4. 严禁：不要在输出里说"我要帮你切换到 X"这类话，切换动作是静默的
  tools:
    allowed: [activate_capability, delegate_to_subagent, list_capabilities]
```

## 四、文档使用规则

### 4.1 各能力文档应该写什么

每份能力文档（01~09）只需要写：

1. **能力概述**（一句话）
2. **本能力的子目录映射**（引用 §1.3，如 "见通用约定 §1.3 表中本行"）
3. **本能力特有的业务流程**（评分维度、特殊提示词、特殊评分规则等）
4. **本能力 Agent 的工具清单与 System Prompt**
5. **本能力依赖的特殊 Tauri 命令**（不在通用工具表里的）
6. **本能力的 UI 进度反馈**（哪些工具调用要显示进度条）

### 4.2 各能力文档不应该再写什么

以下内容**只在本文档维护**，能力文档不再重复：

- ❌ 工作空间根目录路径
- ❌ 完整的目录结构树
- ❌ 通用混合存储策略矩阵
- ❌ 拖拽上传双通道流程
- ❌ `WorkspaceFileTree.tsx` 行为
- ❌ 静默切换准则与 `activate_capability` 实现
- ❌ 跨能力共用工具的接口定义（`list_files`、`export_to_*` 等）

## 五、前端样式约定（CSS 变量隔离规则）

### 5.1 双 Token 体系的冲突风险

本项目同时使用：

| 体系 | 位置 | 作用 |
|---|---|---|
| 项目 Design Token | `src/index.css` 顶部 `@theme {}` 块 | 品牌色、间距、字体等全局 token（**唯一可信源**） |
| shadcn/ui 脚手架变量 | `src/index.css` 底部 `:root {}` / `.dark {}` 块 | shadcn 组件内部依赖的 CSS 变量 |

**冲突根因**：shadcn 每次执行 `npx shadcn add <component>` 都可能在 `index.css` 末尾**追加或覆盖** `:root {}` 块，写入 light mode 白色变量（如 `--background: oklch(1 0 0)`）。由于本项目是 **dark-first**，没有 `class="dark"` 切换，`.dark {}` 里的深色变量永远不会生效，`:root {}` 的白色变量会直接覆盖 `@theme` 里的深色 token，导致**白屏**。

> 这个问题在 Phase D 验收时实际触发，根因已确认。

### 5.2 强制规则

1. **`@theme {}` 是唯一颜色 token 可信源**，所有颜色、间距、字体 token 只在此处定义，不在 `:root {}` 里重复定义颜色。

2. **`:root {}` 只保留 `--radius`**（shadcn 组件圆角依赖），其余 shadcn 生成的颜色变量一律删除：
   ```css
   /* ✅ 正确 */
   :root {
     --radius: 0.625rem;
   }
   
   /* ❌ 禁止 — shadcn 脚手架生成的 light mode 颜色变量，会覆盖 @theme */
   :root {
     --background: oklch(1 0 0);
     --foreground: oklch(0.145 0 0);
     /* ... */
   }
   ```

3. **每次执行 `npx shadcn add` 后必须检查** `src/index.css` 末尾是否被追加了新的 `:root {}` 颜色变量，若有则立即删除。

4. **`.dark {}` 块保留**（shadcn 组件在 `.dark` 类下的样式覆盖），但本项目不依赖它做主题切换，它的存在不影响 dark-first 渲染。

### 5.3 shadcn 组件引入 SOP

```
1. npx shadcn add <component>
2. 检查 src/index.css 末尾是否新增了 :root {} 颜色变量
3. 若有 → 删除所有颜色变量，只保留 --radius
4. 验证页面背景仍为深色（#0F1115）
```

## 六、前端模块隔离规则（Node.js 模块禁止静态引入）

### 6.1 问题根因

`@modelcontextprotocol/sdk/client/stdio.js` 是 **Node.js 专属**传输层，依赖链为：

```
stdio.js → cross-spawn → which → process.env（Node.js 全局）
```

若在前端代码中**静态 import** 此模块，Vite dev 模式下 esbuild 会将其打进浏览器 bundle，导致：

```
ReferenceError: Can't find variable: process → React 崩溃 → 白屏
```

> 注意：`vite.config.ts` 的 `build.rollupOptions.external` **只对 production build 生效**，dev 模式必须同时配置 `optimizeDeps.exclude`。

### 6.2 强制规则

1. **禁止静态 import Node.js 专属模块**（`cross-spawn`、`which`、`node:*`、`@modelcontextprotocol/sdk/client/stdio`）到任何前端文件。

2. **必须使用动态 import**（`await import(...)`）包裹 Node.js 专属模块，让 Vite/esbuild 不将其打进初始 bundle：
   ```ts
   // ✅ 正确 — 动态 import，不污染浏览器 bundle
   const { StdioClientTransport } = await import("@modelcontextprotocol/sdk/client/stdio.js")
   
   // ❌ 禁止 — 静态 import，会拉入 cross-spawn → process → 白屏
   import { StdioClientTransport } from "@modelcontextprotocol/sdk/client/stdio.js"
   ```

3. **`vite.config.ts` 双重防护**：Node.js 模块必须同时出现在两处：
   ```ts
   build: {
     rollupOptions: {
       external: ["cross-spawn", "which", "@modelcontextprotocol/sdk/client/stdio.js"]
     }
   },
   optimizeDeps: {
     exclude: ["cross-spawn", "which", "@modelcontextprotocol/sdk/client/stdio.js"]
   }
   ```

4. **Tauri 架构原则**：本地 MCP server 的 stdio 连接由 **Rust 侧**管理，前端 WebView 只使用 HTTP/SSE transport，永远不应该执行 `StdioClientTransport` 的实际连接逻辑。
